# IDEA MINE 프로젝트 문서 v0.1

## 1. 프로젝트 한 줄 정의
**IDEA MINE**은 AI 기술, 타겟, 산업, 가치, 수익모델을 조합해 기획 아이디어를 발굴하고 발전시키는 **픽셀 아트 기반 아이디어 제너레이션 플랫폼**이다.

---

## 2. 프로젝트 개요

### 2-1. 문제 정의
AI 기술은 빠르게 늘어나고 있지만, 기획자는 다음 문제를 반복해서 겪는다.
- 어떤 기술을 어떤 타겟에게 붙여야 할지 감이 잘 오지 않는다.
- 아이디어가 있어도 비즈니스 가치와 수익 모델까지 연결하기 어렵다.
- 떠오른 아이디어가 일회성 메모로 끝나고 자산화되지 않는다.
- 좋은 조합을 찾기까지 시간이 오래 걸리고, 실패한 시도는 기록되지 않는다.

### 2-2. 해결 방식
IDEA MINE은 아이디어 발상을 **광산 채굴 메타포**로 단순화한다.
- 키워드 탐색 = 채굴
- 조합 실험 = 제련
- 개요서 발전 = 가공
- 공개 전시 = 쇼케이스
- 투자자 연결 = 거래소/매칭

즉, 사용자는 막연한 브레인스토밍이 아니라 **조합 가능한 원석 체계**를 바탕으로 빠르게 아이디어를 만들고, 발전시키고, 저장하고, 보여줄 수 있다.

### 2-3. 핵심 가치 제안
- **빠른 발상:** 빈 화면에서 시작하지 않게 해준다.
- **구조화된 기획:** Tech, Who, Domain, Value, Money를 한 번에 보게 한다.
- **아이디어 자산화:** 생성 → 개요서화 → 보관 → 전시까지 이어진다.
- **차별적 경험:** 픽셀 아트와 광산 세계관으로 감성적 몰입감을 만든다.

## 3. 브랜드/컨셉 정의

### 3-1. 브랜드 키워드

- 팝핑 아이디어
- 데이터 기반 상상력

### 3-2. 톤앤매너
- 단순한 생산성 툴보다 **가볍게 시작하고 깊게 파는** 느낌
- 차갑고 기술적인 AI보다 **귀엽고 영리한 발상 보조자** 느낌
- 말투는 너무 유치하지 않되, 살짝 게임적인 표현 사용
  - 예: 채굴, 제련, 광맥, 금고, 전시, 안전모, 곡괭이

### 3-3. Naming 해석 정리
- **Ownership:** 내 아이디어라는 소유감
- **Excavation:** 깊이 파고드는 탐색
- **Data Mining:** 데이터 기반의 설계 감각
- **Explosive Mine:** 터지는 아이디어
- **M.I.N.E:** Multimodal Insight & Next-generation Evolution

## 4. 사용자 정의

### 4-1. 주요 사용자군
#### A. 일반 기획 입문자
- AI는 흥미롭지만 어떻게 서비스 아이디어로 연결할지 막막한 사용자
- 빠른 추천 조합과 예시 아이디어를 원함

#### B. 실무형 기획자 / PM / 창업 준비자
- AI 기술을 실제 시장 문제와 연결하고 싶은 사용자
- 랜덤 추천보다 직접 조합, 저장, 개요서화가 중요함

#### C. 투자자 / 스카우터
- 특정 시장, 기술, 타겟 조합을 중심으로 유망 프로젝트를 탐색하려는 사용자
- 전시된 요약과 필터링, 알림, 연락 수단이 중요함

## 5. 정보 구조 (세계관 기반)

### 5-1. 공간 구조
#### 1) The Mine
키워드를 발견하는 탐색 공간
- 오늘의 추천 조합
- 랜덤 채굴
- 리롤
- 트렌드 키워드 노출

#### 2) The Lab
키워드를 조합하고 실험하는 공간
- 직접 키워드 선택
- 슬롯머신 조합
- AI 아이디어 생성
- 커스텀 키워드 관리

#### 3) The Vault
아이디어와 개요서를 보관하는 공간
- 저장한 아이디어 목록
- 상세 개요서 보관
- 북마크 / 폴더 / 태그

#### 4) The Showcase
아이디어를 공개 전시하는 공간
- 티저 전시
- 인기 아이디어
- 투자자 노출용 카드

#### 5) The Exchange
아이디어와 투자자를 연결하는 공간
- 투자자 필터
- 관심 키워드 설정
- 매칭/연락 기능

#### 6) The Observatory
상위 지표 및 인사이트를 보는 공간
- 어떤 조합이 많이 생성되는지
- 어떤 가치/기술이 인기인지
- 투자 관심 분포

- 저장 여부

- 투자 관심도 / 조회수 / 반응

## 11. UX 관점에서 중요한 포인트

### 11-1. 강점
- 메타포가 뚜렷해서 기억에 남는다.
- AI 툴의 차가움을 줄이고 감성적인 접근이 가능하다.
- 키워드 조합 구조가 직관적이라 설명하기 쉽다.

### 11-2. 주의점
- 세계관이 너무 강하면 핵심 가치가 흐려질 수 있다.
- 예쁘기만 하고 실제 기획 효율이 떨어지면 재방문이 약해진다.
- 슬롯머신 재미와 실무 생산성의 균형이 중요하다.

### 11-3. UX 원칙 제안
- 항상 “지금 무엇을 해야 하는지”가 명확해야 함
- 게임적 요소는 보조, 핵심은 결과물 품질
- 생성 결과는 예쁜 카드보다 비교/저장/발전이 쉬워야 함

---

## 12. 차별화 포인트

### 일반 AI 아이디어 생성기와 다른 점
1. 단순 프롬프트 입력형이 아니라 **조합 구조가 명확함**
2. 아이디어 생성에서 끝나지 않고 **개요서화/전시/연결**로 이어짐
3. 광산 세계관과 픽셀 아트로 **브랜드 자산화가 쉬움**
4. 실패한 조합도 기록 가능한 **실험 아카이브**로 발전 가능

## 17. 현재 버전 총평
IDEA MINE은 단순한 아이디어 생성기가 아니라,
**조합형 발상 → 개요서 발전 → 전시 → 연결**로 이어질 수 있는 구조를 이미 갖고 있다.

강한 세계관, 명확한 택소노미, 수익 모델의 다층성까지 초안이 잘 잡혀 있다.
앞으로 중요한 것은 **초기 타겟을 좁히고**, **MVP에서 무엇을 반드시 검증할지 선명하게 정하는 것**이다.

## 18. 레거시 문서 반영 원칙 (이전 Idea Bank / Project Vision 기준)

이 프로젝트는 완전히 백지에서 새로 만드는 것이 아니라, 기존 문서의 유효한 가설을 선별적으로 계승하는 방식으로 정리한다.

### 18-1. 그대로 살릴 만한 것
1. **핵심 루프**
   - 구조적 발산 → 선택 → 평가 → 축적
   - 이 루프는 지금의 IDEA MINE에도 그대로 맞는다.

2. **개인 생산성 엔진 관점**
   - “매일 5분 발산” 같은 반복 사용 시나리오는 초기 제품 습관 형성에 유리하다.

3. **Human-in-the-Loop 방향성**
   - 사람이 조합하고, AI가 확장하고, 다시 사람이 선택/평가하는 구조는 유지 가치가 높다.

4. **생성-평가 연결 구조**
   - 단순 생성이 아니라 PRD/개요서/평가까지 연결되는 파이프라인은 중요한 차별점이다.

5. **축적과 재탐색의 중요성**
   - 예전 아이디어를 다시 찾고, 북마크하고, 회고에 활용하는 구조는 반드시 남길 가치가 있다.

### 18-2. 수정해서 가져올 것
1. **브랜드명 / 세계관**
   - 기존 Idea Bank 중심 표현은 IDEA MINE의 광산 메타포로 전면 재해석한다.

2. **타겟 정의**
   - 기존의 “AI 엔지니어 중심”은 더 넓거나 더 정확한 초기 타겟(예: AI PM, 기획자, 창업 준비자)으로 다시 조정할 필요가 있다.

3. **키워드 구조의 경직성**
   - 5개 카테고리 구조는 좋지만, 초기부터 지나치게 고정된 대규모 매트릭스로 시작할지는 다시 검토한다.

4. **용어 체계**
   - Deep Report, PRD 등은 유지 가능하나, 현재 브랜드 톤앤매너와 맞게 재명명할 수 있다.

### 18-3. 과감히 빼도 되는 것
1. 초기 단계에서 과한 범위 확장
   - 운영 문서 전체, 고도화된 규칙 체계, 복잡한 평가 시스템은 MVP 전에는 축소 가능

2. 너무 이른 최적화
   - 중복 통제, 추천 고도화, 회고 자동화 등은 핵심 사용성 검증 이후로 미뤄도 된다.

3. 브랜딩과 어긋나는 표현
   - 생산성 툴 중심 표현이 강해서 세계관 몰입을 해치는 부분은 정리 대상이다.

### 18-4. 현재 기준의 해석
기존 Project Vision은 **제품의 뼈대**로서 여전히 유효하다.
다만, 지금의 IDEA MINE은 그 뼈대 위에
- 더 강한 브랜드 세계관,
- 더 또렷한 감성 UX,
- 더 입체적인 수익 구조,
- 더 확장된 쇼케이스/투자 연결 가능성
을 얹는 방향으로 재구성하는 프로젝트로 본다.

### 18-5. 실무 원칙
앞으로 기존 문서를 다룰 때는 아래 3가지 라벨로 정리한다.
- **KEEP**: 그대로 유지
- **REFRAME**: 의미는 유지하되 이름/표현/구조 변경
- **DROP**: 현재 버전에서는 제거

## 19. 레거시 해체표: KEEP / REFRAME / DROP

기준 문서인 기존 Project Vision을 현재 IDEA MINE 방향으로 다시 분류하면 아래와 같다.

### 19-1. KEEP
#### 1) 핵심 루프
- 구조적 발산 → 선택 → 평가 → 축적
- 이건 제품의 본질에 가깝기 때문에 유지한다.

#### 2) 매일 반복 사용 리듬
- “매일 아침 5분” 같은 짧은 세션 구조는 재방문 습관을 만들기 좋다.
- 다만 시간대는 고정하지 말고, 아침/저녁 데일리 세션 개념으로 확장 가능하다.

#### 3) Human-in-the-Loop
- 사람이 키워드를 고르고,
- AI가 아이디어를 확장하고,
- 다시 사람이 고르는 구조는 계속 가져간다.

#### 4) 생성 → 개요서/평가 연결
- 아이디어 생성만으로 끝내지 않고
- 정리, 저장, 발전으로 이어지는 파이프라인은 유지한다.

#### 5) 축적과 재탐색
- 세션 로그
- 선택/비선택 기록
- 북마크
- 다시 찾기 기능
이 구조는 반드시 남긴다.

### 19-2. REFRAME
#### 1) 브랜드 정체성
- **기존:** Idea Bank / 개인 생산성 엔진 / 인터랙티브 워크벤치
- **변경:** IDEA MINE / 픽셀 아트 광산 / 영감 채굴 게임형 워크벤치

#### 2) 타겟 정의
- **기존:** AI 엔지니어 중심
- **변경:** AI PM, 기획자, 창업 준비자, 실무형 아이디어 메이커 중심

#### 3) 키워드 시스템 표현
- **기존:** 5개 카테고리 × 96 키워드 매트릭스
- **변경:** 5원석 조합 시스템
- 내부 구조는 유지하되, 사용자에게는 표보다 게임적 칩/원석/슬롯 UI로 보여준다.

#### 4) 결과물 명칭
- **기존:** PRD, Deep Report
- **변경 방향:** 개요서, 청사진, 설계도, 쇼케이스 티저 등 브랜드 톤에 맞는 명칭으로 재해석 가능

#### 5) 운영 문서 관점
- **기존:** 규칙 문서 중심
- **변경:** 플레이 규칙 + 추천 로직 + 실험 로그라는 제품 운영 자산으로 해석

### 19-3. DROP
#### 1) 너무 딱딱한 생산성 툴 중심 표현
- 메모 앱, 문서 툴, 일반 업무 툴 같은 느낌이 강한 표현은 걷어낸다.

#### 2) MVP 이전의 복잡한 평가 체계
- 초반부터 정교한 점수 체계, 중복 통제, 고급 추천 시스템은 빼도 된다.

#### 3) 초기에 너무 무거운 운영 구조
- 상세한 규칙 문서 전체 체계화,
- 고도화된 자동 회고,
- 복잡한 분류 체계는 뒤로 미룬다.

#### 4) “엔지니어 전용”으로 좁히는 포지셔닝
- 기술 친화성은 살리되, 제품 자체는 더 넓은 기획형 사용자에게 열어둔다.

### 19-4. 결론
이전 문서는 폐기 대상이 아니라 **제품의 엔진 설계도**로 본다.
지금의 리빌드는 그 엔진 위에
- 게임적 감각,
- 픽셀 아트 정체성,
- 재방문성,
- 전시/연결 구조
를 덧입히는 작업이다.

## 20. 제품 방향성 업데이트: “일처럼 쓰는 게임, 게임처럼 들어오는 기획 툴”

### 20-1. 핵심 방향
IDEA MINE의 핵심 포인트는 단순히 예쁜 픽셀 UI가 아니다.
사용자가 **가볍게 접속하고, 한두 번 더 눌러보고 싶고, 오늘도 광산에 들어가 보고 싶게 만드는 리듬** 자체가 중요하다.

즉, 제품 목표는 아래 2가지를 동시에 만족하는 것이다.
- **게임처럼 진입 장벽이 낮다**
- **실무 툴처럼 결과물은 남는다**

### 20-2. 제품 경험 문장
> IDEA MINE은 “아이디어를 채굴하는 픽셀 도트 게임처럼 보이지만, 실제로는 기획 자산을 쌓아가는 라이트한 워크벤치”다.

### 20-3. 왜 이 방향이 중요한가
일반적인 AI 아이디어 툴은 한 번 쓰고 끝나는 경우가 많다.
반면 IDEA MINE은 아래 요소를 통해 반복 접속 동기를 만들어야 한다.
- 오늘의 광맥이 매번 달라짐
- 리롤이 재미있음
- 결과 카드가 수집 욕구를 줌
- 저장과 성장의 흔적이 남음
- 다음 접속 이유가 생김

### 20-4. 게임화에서 반드시 살릴 요소
#### 1) 짧은 세션
- 30초~5분 안에 한 번의 보상을 느껴야 한다.
- 홈 진입 후 바로 “오늘의 채굴”이 보여야 한다.

#### 2) 즉시 보상
- 조합을 누르면 결과가 바로 튀어나와야 한다.
- 애니메이션, 사운드, 카드 등장 연출이 중요하다.

#### 3) 가벼운 선택의 즐거움
- 슬롯머신처럼 고르기 쉽고
- 너무 많은 입력을 요구하지 않아야 한다.

#### 4) 수집 욕구
- 마음에 드는 아이디어를 금고에 저장
- 좋은 조합을 세트로 보관
- 내 채굴 히스토리가 쌓여야 한다.

#### 5) 성장 감각
- 내가 더 잘 캐고 있다는 느낌
- 내 취향에 맞는 광맥을 찾는 느낌
- 내 금고가 점점 채워지는 느낌

### 20-5. 게임화에서 조심할 것
#### 1) 장식만 게임처럼 보이는 것
- 픽셀 그래픽만 있고 실제 루프가 재미없으면 오래 못 간다.

#### 2) 너무 복잡한 게임 규칙
- 진짜 게임처럼 복잡해지면 기획 툴로서 피곤해진다.

#### 3) 결과물 품질이 약해지는 것
- 재미 때문에 결과가 얕아지면 실무 유저가 떠난다.

#### 4) 광고/과금이 몰입을 깨는 것
- 리듬을 끊는 순간 재방문성이 크게 떨어질 수 있다.

### 20-6. 그래서 MVP에서 검증해야 할 것
MVP는 “게임 세계관 전체”보다 아래를 먼저 검증해야 한다.

#### A. 오늘의 채굴이 실제로 다시 들어오고 싶게 만드는가?
- 데일리 추천 조합
- 리롤 재미
- 결과 카드 만족도

#### B. 저장과 누적이 실제로 의미 있게 느껴지는가?
- 금고 저장
- 다시 보기
- 내가 고른 취향 반영

#### C. 한 번의 생성이 너무 무겁지 않은가?
- 10개 생성이 좋은지,
- 3개 고품질 카드가 더 좋은지,
- 실험이 필요하다.

### 20-7. 제안하는 초기 제품 정의
초기 버전의 정의는 이렇게 잡는 것이 가장 자연스럽다.

> 픽셀 광산 세계관 속에서 AI 서비스 아이디어를 가볍게 채굴하고, 마음에 드는 결과를 금고에 쌓아가는 게임형 기획 도구

이 정의는
- 감성,
- 반복 접속,
- 실무 효용
을 동시에 설명한다.

## 22. 1차 제품 설계 정리 (현재 합의안)

### 22-1. 먼저 잡아야 할 것: 메뉴 구조
사용자 의견 기준으로, 현재 가장 먼저 확정해야 하는 것은 **메뉴 구성**이다.
이유는 IDEA MINE이 단순 문서 툴이 아니라 **게임처럼 빠르게 들어와 바로 행동하게 만드는 구조**여야 하기 때문이다.

핵심 원칙은 아래와 같다.
- 메뉴는 적어야 한다.
- 첫 화면에서 바로 채굴이 가능해야 한다.
- 저장/보상/퀘스트가 한눈에 보여야 한다.
- 세계관은 살리되 길을 잃게 만들면 안 된다.

### 22-2. 추천 MVP 메뉴 구조
초기에는 4개 메인 메뉴가 가장 적절하다.

#### 1) 홈 / 광산 (The Mine)
가장 중요한 첫 화면.
- 오늘의 랜덤 키워드 조합 3개
- 리롤
- 연속 접속 보상
- 데일리 퀘스트
- 주말 특별 광맥 진입

이 메뉴의 역할은 “들어오면 바로 한 판 시작”이다.

#### 2) 실험실 / 믹스 (The Lab)
직접 조합하고 실험하는 공간.
- 키워드 커스텀 선택
- Premium 전용 자유 조합
- Tech 보장 아이템 사용
- 희귀 조합 탐색

이 메뉴는 무료보다 프리미엄 가치를 느끼게 하는 핵심 공간이다.

#### 3) 금고 (The Vault)
결과를 저장하고 다시 보는 공간.
- 저장한 아이디어
- 선택한 프로젝트 개요 문서
- 즐겨찾기 조합
- 과거 채굴 기록
- 금고 확장 슬롯

이 메뉴는 “이 서비스가 남는 툴”이라는 감각을 만든다.

#### 4) 광부수첩 / 마이 (My Mine)
개인 진행 상황과 꾸미기 요소를 모으는 공간.
- 프로필
- 연속 접속 현황
- 보유 재화
- 장비/스킨
- 업적
- 설정
- 구독 관리

초기에는 상점도 이 메뉴 안에 넣는 것이 단순하다.

### 22-3. 지금은 메인 메뉴에서 빼는 것이 좋은 것
초기에는 아래 항목은 탭으로 독립시키지 않는 것이 좋다.

#### A. 쇼케이스
좋은 기능이지만 초기에는 홈 배너나 금고 내부 섹션으로 넣어도 된다.
독립 메뉴가 되면 제품이 무거워진다.

#### B. 투자자 연결 / 거래소
장기 확장 포인트로는 좋지만, MVP 메인 흐름에 넣으면 포커스가 흔들린다.
초기에는 숨겨두거나 추후 별도 모드로 분리하는 것이 낫다.

#### C. 전망대 / 분석 대시보드
후반 확장용으로는 좋다. 하지만 초반에 메인 탭으로 올리면 게임 리듬이 죽는다.

## 23. 추천 핵심 세션 루프

### 23-1. 기본 플레이 흐름
사용자 의견을 반영한 현재 가장 자연스러운 루프는 아래와 같다.

1. 홈 진입
2. 오늘의 랜덤 키워드 조합 3개 확인
3. 마음에 드는 조합 1개 선택
4. 선택한 조합 기반으로 아이디어 draft 10개 생성
5. 그중 1개 선택
6. 선택한 1개에 한해서 프로젝트 개요 문서 생성
7. 저장 또는 종료
8. 보상/퀘스트/다음 채굴 동기 확인

### 23-2. 왜 이 구조가 좋은가
- 30초 안에 “뭔가 나왔다”는 보상이 있다.
- 처음부터 1개만 뽑는 것보다 실패 방어가 된다.
- 아이디어 10개는 발산용,
- 프로젝트 개요 문서는 수렴용으로 역할이 다르다.
- 무료/광고/구독 과금 포인트를 자연스럽게 넣을 수 있다.

### 23-3. 핵심 UX 포인트
이 루프에서 중요한 것은 “10개가 많다”가 아니라,
**10개가 빠르게 훑어볼 수 있어야 한다**는 점이다.

따라서 아이디어 draft는 아래처럼 가벼워야 한다.
- 제목 1줄
- 설명 2~3줄
- 핵심 타겟/가치 태그
- 빠른 저장/비저장

개요 문서는 반대로 한 단계 더 진지하게 들어가야 한다.
- 문제 정의
- 타겟 유저
- 핵심 기능
- 시장성/BM 초안
- 차별점
- MVP 제안

## 24. 무료 / 광고 / 구독 구조 초안

### 24-1. 기본 원칙
광고와 과금은 **핵심 루프를 방해하지 않고, 더 깊게 파고 싶은 순간**에 자연스럽게 나와야 한다.
즉,
- “시작하려면 결제하세요”는 나쁘고
- “한 번 더 파고 싶다면 열어드릴게요”는 자연스럽다.

### 24-2. 무료 유저 기본 제공안
- 매일 랜덤 키워드 조합 3개 제공
- 무료 리롤 2회
- 하루 1회 아이디어 draft 10개 생성
- 하루 1회 프로젝트 개요 문서 생성

이 정도면 체험은 충분하고, 더 쓰고 싶은 순간이 생긴다.

### 24-3. 광고를 넣기 좋은 지점
#### A. 리롤 소진 직후
- “곡괭이 수리 완료! 5회 추가 리롤”
- 가장 자연스러운 보상형 광고 지점

#### B. 하루 생성 한도 소진 직후
- “야간 작업권 획득”
- 추가 아이디어 draft 10개 생성 1회 제공

#### C. 개요 문서 추가 생성 직전
- “정밀 스캔권” 같은 연출로 광고 1회 시청 후 추가 생성 제공 가능

### 24-4. 구독자 혜택이 명확해야 하는 부분
구독은 단순히 광고 제거가 아니라 **게임을 내 방식대로 더 깊게 플레이하는 권한**처럼 보여야 한다.

예시:
- 자유 조합 무제한
- 커스텀 키워드 관리
- 프로젝트 개요 문서 다회 생성
- 금고 저장 슬롯 확대
- 희귀 광맥 조기 해금
- 쇼케이스 우선 노출
- 광고 없이 연속 플레이

## 27. 홈 화면 상세 설계 v1

현재 기준에서 홈 화면은 IDEA MINE의 가장 중요한 공간이다.
이 화면의 목표는 아래 3가지를 동시에 달성하는 것이다.
- 3초 안에 오늘 할 일이 보인다.
- 30초 안에 첫 결과를 얻을 수 있다.
- 다시 들어오고 싶을 이유가 남는다.

### 27-1. 홈 화면 전체 구조
1. 상단 상태바
2. 오늘의 광맥 3개
3. 리롤 버튼
4. 데일리 퀘스트
5. 주말 특별 광맥 배너
6. 금고 미리보기
7. 하단 예고 구역 (공사 중 콘텐츠)

### 27-2. 상단 상태바
상단 상태바는 정보가 많아 보이면 안 된다.
핵심은 “내 현재 상태”를 한눈에 보여주는 것이다.

#### 꼭 들어갈 요소
- 프로필 아이콘 / 닉네임
- 연속 접속 일수
- 보유 재화 (Pink Diamond)
- 알림 아이콘
- 설정 진입

#### 선택적으로 넣을 요소
- 오늘 남은 무료 채굴 횟수
- 현재 등급/광부 레벨

#### 추천 표현 예시
- `도민 광부님 ⛏️`
- `연속 채굴 4일째`
- `💎 120`

#### UX 원칙
- 숫자는 작고 명확하게
- 상단은 조작보다 상태 확인 중심
- 너무 많은 정보는 마이페이지로 넘긴다

### 27-3. 오늘의 광맥 3개
이 영역은 홈의 핵심 CTA다.
사용자는 여기서 바로 하나를 눌러야 한다.

#### 카드 1장에 들어갈 정보
- 광맥 이름 (자동 생성 별칭)
- 키워드 조합 3~5개
- 분위기 태그 1개
- 희귀도 또는 추천도 배지 1개
- 선택 버튼 또는 카드 전체 클릭

#### 추천 카드 구조
- 상단: 광맥 이름
- 중앙: 키워드 칩 3~5개
- 하단: 분위기 태그 + 희귀도

#### 광맥 이름 예시
- `감정 케어 광맥`
- `실버 헬스 광맥`
- `혼자 사는 사람의 자동화 광맥`
- `멀티모달 생산성 광맥`

#### 분위기 태그 예시
- `실용형`
- `트렌디`
- `실험적`
- `시장성 높음`
- `감성 특화`

#### 희귀도 예시
- 일반 광맥
- 반짝 광맥
- 희귀 광맥

#### UX 원칙
- 키워드는 읽기 쉬워야 한다.
- 카드마다 성격이 달라 보여야 한다.
- 너무 정보가 많으면 클릭 전 피로해진다.

### 27-4. 리롤 버튼
리롤은 단순 새로고침이 아니라 홈의 핵심 재미 장치다.
따라서 버튼이 숨어 있으면 안 된다.

#### 권장 위치
- 오늘의 광맥 3개 바로 아래 중앙
- 또는 카드 영역 우측 상단 보조 버튼

#### 추천 메인 문구
- `다시 파기`
- `광맥 새로 찾기`
- `곡괭이 다시 휘두르기`

#### 보조 정보
- 남은 무료 리롤 횟수 표시
- 예: `오늘 무료 2/2`

#### 리롤 연출
- 카드가 바로 바뀌기보다
- 짧은 셔플 애니메이션 후 새 광맥 등장
- 사운드가 있다면 가벼운 “딸깍-반짝” 계열 추천

### 27-5. 데일리 퀘스트
데일리 퀘스트는 사용자가 홈에서 바로 한 번 더 행동하게 만드는 보조 루프다.
핵심 채굴보다 더 크면 안 되고, 자연스러운 보조 동기여야 한다.

#### 초기 추천 퀘스트 유형
- 오늘 광맥 1회 선택하기
- 아이디어 1개 금고에 저장하기
- 리롤 1회 사용하기
- 프로젝트 개요 1회 생성하기

#### 보상 예시
- Pink Diamond 소량
- 추가 리롤 1회
- 금고 확장 조각
- 희귀 광맥 확률 증가권

#### UI 방식
- 카드 1장 안에 오늘의 퀘스트 1~3개 노출
- 전부 펼치지 말고 핵심만 보여주기

### 27-6. 주말 특별 광맥 배너
이 영역은 이벤트성과 기대감을 만든다.
평일에는 티저, 주말에는 활성 배너로 운영할 수 있다.

#### 평일 문구 예시
- `주말 특별 광맥이 준비되고 있어요`
- `토요일 00:00, 희귀 광맥 개방 예정`

#### 주말 문구 예시
- `이번 주말 한정: 미래 직업 광맥`
- `희귀 멀티모달 광맥 개방 중`

#### UX 원칙
- 화면을 너무 차지하면 안 된다.
- 하나의 강한 배너면 충분하다.
- 주말마다 테마가 달라야 반복 접속 이유가 생긴다.

### 27-7. 금고 미리보기
금고 미리보기는 “이 앱에 내 결과가 쌓인다”는 감각을 주는 역할이다.
홈에서 너무 크게 노출할 필요는 없지만, 반드시 있어야 한다.

#### 추천 구성
- 최근 저장한 아이디어 1~3개 미리보기
- 최근 개요 문서 1개 미리보기
- `금고 열어보기` 버튼

#### 표시 정보 예시
- 제목
- 생성 날짜
- 관련 키워드 1~2개
- 상태 배지 (`draft`, `개요서 완성`)

#### UX 원칙
- 이 영역은 자랑보다 회수 장치다.
- “지난번에 캐던 것 다시 보기” 느낌이 중요하다.

### 27-8. 하단 예고 구역
정식 메뉴로 올리지 않은 기능은 홈 하단에서 세계관형 예고 구역으로 보여줄 수 있다.

#### 예시 문구
- `🚧 광산 확장 공사 중`
- `🔒 일정 광맥 깊이 도달 시 개방 예정`
- `📡 관측 장비 보정 중`

#### 적용 대상
- 쇼케이스
- 거래소
- 전망대

이 방식은 기능 부재를 약점처럼 보이게 하지 않고,
세계관 속 확장 포인트처럼 느끼게 해 준다.

### 27-9. 홈 화면 핵심 원칙 요약
- 첫 화면은 무조건 “바로 채굴” 중심
- 상태 정보는 가볍게
- 카드 선택은 명확하게
- 보상과 저장은 보조 루프로 배치
- 공사 중 기능은 기대감만 남기고 복잡성은 줄이기

## 28. 오늘의 광맥 카드 상세 설계 v1

사용자 의견을 반영하여, 오늘의 광맥 카드는 아래 원칙으로 설계한다.

### 28-1. 카드 설계 원칙
- **키워드 5개를 모두 보여준다.**
- 카드는 **가로 슬라이드(캐러셀)** 형태로 넘기며 선택한다.
- 각 카드에는 **희귀도 / 추천도**를 넣는다.
- 카드 문구는 너무 게임스럽기만 하지 않고, **기획 툴로서 이해 가능한 수준의 명확함**을 유지한다.

### 28-2. 카드 1장 기본 구조
#### 상단
- 배지 1: 추천도 또는 희귀도
- 배지 2: 분위기 태그

예:
- `오늘의 추천`
- `희귀 광맥`
- `시장성 높음`
- `실험적`

#### 중앙
- 광맥 이름
- 한 줄 설명

예:
- `멀티모달 실버 케어 광맥`
- `실버 세대의 일상 케어를 멀티모달 AI와 구독형 모델로 풀어보는 조합`

#### 본문
- 5원석 키워드 전부 노출
  - Who
  - Domain
  - Tech
  - Value
  - Money

예:
- `Who | 실버 세대`
- `Domain | 헬스케어`
- `Tech | 멀티모달`
- `Value | 시간 단축`
- `Money | 구독형`

#### 하단
- 결과 예고 문구
- CTA 버튼

예:
- `이 조합으로 아이디어 draft 10개를 바로 채굴할 수 있어요.`
- 버튼: `이 광맥 파기`

### 28-3. 왜 5개를 다 보여주는가
IDEA MINE은 단순 랜덤 카드 뽑기 게임이 아니라,
**조합 자체가 핵심 가치**인 아이디어 발굴 도구다.

따라서 카드에서 키워드를 일부만 보여주면
- 사용자가 왜 이 조합을 고르는지 불분명해지고,
- 저장/비교/학습 가치가 떨어진다.

그래서 5개 키워드는 모두 보여주되,
보이기만 많이 보이는 것이 아니라 **칩/행 구조로 정돈된 형태**여야 한다.

### 28-4. 카드 문구 톤 방향
카드 문구는 아래 3가지 스타일 중 **중간형**이 가장 적합하다.

#### A. 너무 기능적인 스타일
- `키워드 조합 기반 아이디어 생성`
- `이 조합으로 draft 10개 생성`

장점: 명확함
단점: IDEA MINE만의 개성이 약함

#### B. 너무 게임적인 스타일
- `희귀 광맥 발견! 지금 바로 곡괭이를 휘둘러 보세요!`
- `반짝이는 광맥이 당신을 기다립니다!`

장점: 세계관 강함
단점: 실무 툴로서 신뢰감이 약해질 수 있음

#### C. 추천 스타일: 게임성 + 명확함의 균형
- `오늘 유망한 조합이에요.`
- `이 광맥으로 아이디어 10개를 채굴해보세요.`
- `마음에 드는 초안을 하나 골라 개요서로 발전시킬 수 있어요.`

이 방향이 MVP에 가장 적합하다.

### 28-5. 카드 문구 추천 세트

#### 세트 1. 가장 기본형
- 배지: `오늘의 추천`
- 한 줄 설명: `지금 바로 아이디어를 뽑기 좋은 안정적인 조합`
- 하단 문구: `이 조합으로 아이디어 draft 10개를 생성할 수 있어요.`
- 버튼: `아이디어 캐기`

#### 세트 2. 세계관 강조형
- 배지: `반짝 광맥`
- 한 줄 설명: `채굴 가치가 높은 조합입니다.`
- 하단 문구: `이 광맥을 파면 아이디어 draft 10개를 얻을 수 있어요.`
- 버튼: `이 광맥 파기`

#### 세트 3. 실무 균형형
- 배지: `시장성 높음`
- 한 줄 설명: `타겟, 기술, 수익모델의 균형이 좋은 조합`
- 하단 문구: `초안 10개를 빠르게 비교하고, 1개를 개요서로 발전시켜 보세요.`
- 버튼: `초안 뽑기`

### 28-6. 현재 추천 결론
초기 MVP에서는 카드 문구를 아래 방향으로 통일하는 것이 가장 좋다.

#### 추천 규칙
- 제목: 세계관형 이름 사용
- 설명: 실무적으로 이해 가능한 한 줄
- 배지: 추천도/희귀도/분위기 중 1~2개
- CTA: 짧고 명확하게

#### 추천 CTA 우선순위
1. `이 광맥 파기`
2. `아이디어 캐기`
3. `초안 뽑기`

가장 IDEA MINE다운 기본 CTA는 `이 광맥 파기`이고,
가장 직관적인 보조 표현은 `아이디어 draft 10개 생성`이다.

즉, 문구는 아래처럼 조합하는 것이 가장 자연스럽다.
- 상단 배지: `오늘의 추천`
- 제목: `멀티모달 실버 케어 광맥`
- 설명: `실버 세대의 일상 문제를 멀티모달 AI와 구독형 모델로 풀어보는 조합`
- 본문: 5원석 키워드 전체 노출
- 하단 문구: `이 광맥으로 아이디어 draft 10개를 바로 채굴할 수 있어요.`
- 버튼: `이 광맥 파기`

## 29. 핵심 개념 수정: 광석은 '강제 스펙'이 아니라 '투입 재료'다

사용자 피드백 기준으로, IDEA MINE의 키워드 조합은 **결과물에 반드시 전부 반영되어야 하는 확정 스펙**이 아니다.

즉,
- Who / Domain / Tech / Value / Money 광석을 투입하지만,
- 생성된 아이디어는 그중 일부를 강하게 쓰고,
- 일부는 약하게 반영하거나,
- 경우에 따라 직접적으로 드러나지 않을 수도 있다.

이 점은 매우 중요하다. 왜냐하면 IDEA MINE의 핵심은
**정답 조합 생성기**가 아니라,
**원석을 넣고 가능성 있는 아이디어 draft를 발굴하는 채굴 시스템**이기 때문이다.

### 29-1. 개념 정리
- 키워드 = 확정 요구사항이 아니라 **탐색 재료**
- 결과 아이디어 = 재료를 바탕으로 나온 **해석 결과물**
- 따라서 생성 품질의 핵심은 “5개를 모두 기계적으로 반영”이 아니라,
  “조합의 방향성을 얼마나 흥미롭고 설득력 있게 살리느냐”에 있다.

### 29-2. 제품 관점에서의 의미
이 정의를 택하면 제품이 더 유연해진다.

#### 장점
1. 더 자연스러운 아이디어가 나온다.
2. 억지로 5개를 다 끼워 넣는 부자연스러움을 줄일 수 있다.
3. 사용자는 “조합 실험”을 한다는 느낌을 더 받게 된다.
4. draft 10개가 서로 다른 해석을 보여줄 수 있다.

### 29-3. 따라서 카드와 결과의 관계는 이렇게 본다
- 카드에는 **이번 채굴에 투입된 광석**을 보여준다.
- 하지만 결과 아이디어는 **그 광석을 100% 의무 반영한 답안**이 아니다.
- 오히려 각 draft가 어떤 광석을 강하게 살렸는지 차이가 날 수 있다.

### 29-4. 문서 표현 수정 원칙
앞으로 문서에서 키워드 조합을 설명할 때는 아래처럼 표현한다.

#### 수정 전 느낌
- `이 5개 키워드를 모두 반영한 아이디어`

#### 수정 후 권장 표현
- `이 광석 조합을 바탕으로 해석한 아이디어 draft`
- `투입된 광석을 중심으로 발굴한 10개의 가능성`
- `같은 광맥에서도 서로 다른 결을 가진 초안`

### 29-5. 현재 기준의 서비스 정의 보정
IDEA MINE은
**정해진 키워드를 조립해 정답을 출력하는 도구**가 아니라,
**선택한 광석 조합을 바탕으로 여러 방향의 아이디어 초안을 발굴하고, 그중 유망한 하나를 발전시키는 게임형 기획 플랫폼**으로 정의한다.

## 30. 아이디어 생성 파이프라인 재정의: 완성형 10개가 아니라 Raw Seed 발굴

사용자 설명 기준으로, IDEA MINE의 생성 흐름은 기존의 “완성형 아이디어 10개를 바로 생성” 방식에서 벗어난다.

기존 방식은 처음부터
- 제목,
- 요약,
- 설명,
- 구조화된 완성안
을 한 번에 만들려 했기 때문에,
모델이 상대적으로 **안전하고 무난한 답**으로 수렴하기 쉬웠다.

따라서 현재 방향은 아래처럼 바뀐다.

### 30-1. 새 방향의 핵심
초기 단계에서는 완성도를 일부러 낮추고,
대신 **발상 폭과 조합 탐색 범위**를 넓힌다.

즉, 1차 생성의 목표는 “정답 같은 아이디어”가 아니라,
**가능성 있는 아이디어 후보군을 넓게 확보하는 것**이다.

### 30-2. 단계 비유
- **1차: 원석 채굴**
- **2차: 감정하고 고르기**
- **3차: 보석처럼 다듬어 보여주기**

이 비유는 IDEA MINE의 세계관과도 매우 잘 맞는다.

### 30-3. 단계별 정의

#### 1차 단계 = Raw Seed 생성
이 단계에서 모델이 만드는 것은 완성형 아이디어가 아니라 **raw seed**다.

raw seed의 목표:
- 조합을 여러 방향으로 해석해 보기
- 안전한 정답보다 가능성 있는 갈래를 많이 확보하기
- 서로 결이 다른 후보를 뽑아내기

즉, 이 단계는 **발산**이 핵심이다.

#### 2차 단계 = 감정 / 선택
사용자는 raw seed들을 보고,
- 끌리는 방향,
- 시장성 있어 보이는 방향,
- 더 파보고 싶은 방향
을 고른다.

이 단계의 핵심은 **선택**이다.

#### 3차 단계 = 개요 문서 / 정제
선택된 seed 하나를 바탕으로,
- 문제 정의,
- 타겟,
- 핵심 기능,
- 차별점,
- BM,
- MVP
등을 붙여 더 정돈된 결과물로 만든다.

이 단계는 **수렴과 정제**가 핵심이다.

### 30-4. 왜 이 구조가 중요한가
이 구조를 택하면 다음 장점이 생긴다.

1. 모델이 초반부터 모범답안처럼 굳어지지 않는다.
2. 같은 광석 조합에서도 서로 다른 방향의 후보가 나온다.
3. 사용자가 고르는 재미와 개입 가치가 커진다.
4. IDEA MINE의 게임 메타포와 생성 로직이 자연스럽게 연결된다.

### 30-5. Raw Seed의 권장 정보 단위
raw seed는 너무 완성형이면 안 되고,
너무 추상적이어도 선택하기 어렵다.

따라서 초기 권장 구성은 아래 정도가 적절하다.
- **seed 한 줄 이름** 또는 임시 라벨
- **핵심 발상 한 줄**
- **어떤 결의 아이디어인지 보여주는 짧은 설명**
- **강하게 반영된 광석 축 1~2개**

즉, raw seed는 “바로 제출 가능한 기획안”이 아니라,
**발상 단서가 되는 조각**이어야 한다.

### 30-6. 제품 문구 관점의 의미
이 정의를 따르면 결과 화면에서 “아이디어 10개”라고 말하기보다,
상황에 따라 아래 같은 표현이 더 적합할 수 있다.
- `아이디어 seed 10개`
- `가능성 초안 10개`
- `원석 draft 10개`
- `채굴된 발상 조각 10개`

초기 톤앤매너 기준으로는,
- 너무 기술적이면 딱딱하고,
- 너무 시적이면 이해가 어려울 수 있으므로,
**“초안”과 “원석”의 중간 표현**을 찾는 것이 좋다.

### 30-7. 현재 기준의 제품 정의 보강
IDEA MINE은 처음부터 정답처럼 완성된 서비스 아이디어를 내놓는 도구가 아니라,
**광석 조합을 바탕으로 raw seed를 발굴하고, 그중 유망한 하나를 골라 정제해 가는 게임형 아이디어 탐색 플랫폼**으로 본다.

## 31. 다음 논의 우선순위: 이제 무엇을 결정해야 하는가

현재까지 합의된 핵심은 아래와 같다.
- 홈에서는 빠르게 채굴을 시작한다.
- 오늘의 광맥 3개 중 하나를 고른다.
- 그 광맥을 바탕으로 **아이디어 원석 10개**를 만든다.
- 각 원석은 완성형 기획안이 아니라, **핵심 아이디어 한 줄** 중심의 raw seed다.
- 그중 하나를 골라 프로젝트 개요 문서로 발전시킨다.

이제 다음으로 상의해야 할 것은 “원석을 어떻게 고르고, 어떻게 다듬고, 어떤 기준으로 좋은 경험을 만들 것인가”다.

### 31-1. 우선순위 1: 아이디어 원석 1개의 정보 구조
가장 먼저 정해야 할 것.

질문:
- 원석 1개는 어떤 정보까지 보여줄까?
- 너무 빈약하면 고르기 어렵고,
- 너무 길면 다시 완성형 아이디어처럼 무거워진다.

현재 후보:
- 임시 이름
- 핵심 아이디어 한 줄
- 짧은 보조 설명 1줄
- 강하게 반영된 광석 축 1~2개
- 방향성 태그

### 31-2. 우선순위 2: 원석 10개의 다양화 규칙
같은 광맥에서 나온 10개가 서로 얼마나 달라야 하는지 정해야 한다.

질문:
- 실용형 / 감성형 / B2B형 / 소비자형 / 실험형처럼 결을 나눌 것인가?
- 시장성, 참신성, 구현 난이도 등 기준을 분산할 것인가?
- 일부는 안전한 조합, 일부는 튀는 조합으로 의도적으로 섞을 것인가?

### 31-3. 우선순위 3: 사용자가 원석 하나를 고르는 기준
“왜 이걸 골라서 개요서로 보낼까?”의 기준이 필요하다.

질문:
- 제일 끌리는 것?
- 제일 현실적인 것?
- 제일 시장성이 있어 보이는 것?
- 제일 새로워 보이는 것?

이 기준이 정해져야 선택 UX와 카피도 정리된다.

### 31-4. 우선순위 4: 개요 문서의 최소 구조
원석에서 개요 문서로 넘어갈 때 무엇이 추가되는지 정의해야 한다.

질문:
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM
- MVP 범위
중 어떤 항목이 최소인가?

### 31-5. 우선순위 5: 추천도 / 희귀도의 의미
지금 카드에 추천도와 희귀도를 넣기로 했으니,
그게 정확히 무엇을 의미하는지도 정해야 한다.

질문:
- 추천도는 균형감? 트렌드성? 사용자 취향 적합도?
- 희귀도는 흔치 않은 조합? 실험성? 발견 난이도?

### 31-6. 우선순위 6: 과금 지점의 감정 설계
이미 광고와 구독 위치는 대략 잡혀 있다.
이제는 “언제 결제를 떠올리게 하는가”의 감정 설계가 필요하다.

질문:
- 더 파고 싶을 때?
- 더 저장하고 싶을 때?
- 더 정밀하게 보고 싶을 때?

### 31-7. 추천 다음 순서
가장 자연스러운 다음 논의 순서는 아래와 같다.
1. 아이디어 원석 1개의 정보 구조
2. 원석 10개의 다양화 규칙
3. 원석 선택 후 개요 문서의 최소 구조
4. 추천도 / 희귀도 정의
5. 과금/광고 감정 설계

## 32. 아이디어 원석 1개 정보 구조 (확정안)

### 32-1. 기본 구성
- 원석 이름
- 핵심 아이디어 한 줄
- 보조 설명 1줄
- 강하게 반영된 광석 축 1~2개

### 32-2. 목적

- 아직 완성형 기획안처럼 무겁지 않다.
- 하나를 골라 개요 문서로 발전시키기에 충분한 단서를 준다.

### 32-3. 예시 포맷
- **원석 이름:** 혼자 사는 사람의 감정 루틴 도우미
- **핵심 아이디어 한 줄:** 외로움이 심해지는 시간대에 맞춰 대화와 생활 루틴 관리를 함께 제공하는 AI 케어 서비스
- **보조 설명:** 감정 케어와 습관 형성을 묶어 재방문 빈도를 높이는 방향의 초안
- **강한 축:** Value, Tech
- **방향성 태그:** 감성형

## 34. 유료화 전략 제안: 개요 문서는 강하게, 실행 설계는 프리미엄으로

사용자 의견 기준으로, 프로젝트 개요 문서의 기본 퀄리티는 처음부터 높게 가져가는 것이 좋다.
대신 **실행 설계의 깊이**에서 무료와 유료를 나누는 방향이 적절하다.

### 34-1. 왜 이 전략이 좋은가
무료 개요 문서의 품질이 너무 낮으면 아래 문제가 생긴다.
- 서비스의 실력 자체가 약해 보인다.
- 사용자가 “이걸 굳이 저장할 이유가 있나?”라고 느낀다.
- 유료 전환 이전에 신뢰를 잃는다.

반대로 기본 개요 문서의 품질이 높으면 아래 장점이 있다.
- 사용자가 서비스의 핵심 가치를 바로 체감한다.
- 저장, 공유, 재방문 동기가 생긴다.
- “여기서 더 자세히 보고 싶다”는 자연스러운 유료 욕구가 생긴다.

즉,
**무료는 얕게 주는 것이 아니라, 충분히 좋게 주고, 더 실행 가능한 설계에서 업셀링하는 구조**가 더 낫다.

### 34-2. 추천 무료 범위
무료 개요 문서는 아래 수준까지는 충분히 좋아야 한다.
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM 초안
- MVP 범위 (짧고 명확한 1차 버전)

여기서의 MVP 범위는 “무엇을 먼저 만들지” 정도의 방향 제시까지 포함한다.
하지만 세부 우선순위, 화면 단위 정의, 단계별 실행 계획까지는 들어가지 않는다.

### 34-3. 프리미엄으로 유료화하기 좋은 구간
유료는 단순히 더 긴 문서가 아니라,
**바로 실행 가능한 수준으로 내려오는 설계 깊이**를 파는 것이 좋다.

추천 유료 확장 항목:
- 상세 MVP 기능 우선순위
- Phase 1 / 2 / 3 로드맵
- 사용자 플로우 초안
- 화면/기능 단위 구현 범위
- 핵심 가설 및 검증 항목
- 성공 지표 초안
- 출시 순서 제안
- 리스크와 우선 대응 포인트

즉,
무료는 **좋은 기획 개요**,
유료는 **실행 가능한 제품 설계 초안**으로 구분한다.

### 34-4. 추천 티어 구조
#### 무료
- 아이디어 원석 탐색
- 개요 문서 1회 생성
- 고품질 기본 개요 제공

#### 광고 보상
- 개요 문서 추가 생성
- 다른 원석 재시도

#### 구독 / 유료 심화
- 확장 개요 문서
- 상세 MVP 설계
- Phase별 실행 로드맵
- 기능 우선순위와 실험 계획
- 저장/비교/재편집 기능 확장

### 34-5. 사용자에게 어떻게 느껴져야 하는가
이 구조에서 유료는 “숨겨진 기본 기능 해제”처럼 보이면 안 된다.
유저가 느껴야 하는 감정은 아래에 가깝다.
- 무료: `오, 이거만으로도 꽤 괜찮다.`
- 유료: `이건 실제로 실행하려는 사람에게 필요한 깊이다.`

즉, 유료 포인트는 벌칙이 아니라 **실행 해상도 업그레이드**여야 한다.

### 34-6. 추천 결론
현재 IDEA MINE에는 아래 전략이 가장 잘 맞는다.

- **기본 개요 문서 품질은 높게 유지한다.**
- **대신 상세 MVP 범위와 phase 설계, 실행 로드맵은 유료화한다.**

이 방향은
- 제품 신뢰를 지키고,
- 무료 사용자도 만족시키며,
- 실제로 진지한 기획자/창업 준비자에게 명확한 결제 이유를 만든다.

## 35. 아이디어 원석 10개의 다양화 규칙 v1

IDEA MINE에서 다양화 규칙은 단순히 “서로 다르게 보여주기”가 아니다.
핵심은 **같은 광석 조합을 여러 결로 해석하게 만드는 것**이다.

즉,
- 중복은 줄이고,
- 안전한 답만 반복되지 않게 하면서,
- 그래도 선택한 광석과 완전히 무관해지지는 않도록
균형을 잡아야 한다.

### 35-1. 기본 원칙
1. **완전 랜덤이 아니라 통제된 다양성**
   - 엉뚱하기만 한 결과는 줄이고,
   - 서로 다른 방향성을 의도적으로 확보한다.

2. **광석 조합과의 연결은 남긴다**
   - 10개가 전부 다른 서비스처럼 보이면 안 된다.
   - 모두 같은 광맥에서 나온 원석이라는 느낌은 유지해야 한다.

3. **완성도보다 해석 폭을 우선한다**
   - 1차 단계는 polished answer가 아니라
   - “이런 방향도 있네”를 만드는 단계다.

4. **중복 제거는 필수다**
   - 표현만 다르고 사실상 같은 아이디어는 줄여야 한다.

### 35-2. 추천 다양화 구조: 10개를 4개 군으로 나눈다
원석 10개는 아래처럼 성격이 다른 묶음으로 구성하는 것이 좋다.

#### A. 안정형 3개
가장 광석 조합에 가깝고,
현실적으로 이해하기 쉬운 방향.

역할:
- 사용자가 바로 이해할 수 있는 후보 제공
- 너무 실험적이기만 한 결과를 방지

#### B. 확장형 3개
광석 중 일부 축을 강하게 키워
다른 문제 정의나 사용 장면으로 확장한 방향.

역할:
- 같은 조합 안에서 더 넓은 가능성 제시
- 발상 폭 확보

#### C. 전환형 2개
서비스 형태나 BM, 사용 맥락을 바꿔
의외성을 만드는 방향.

역할:
- 단순 반복을 막고
- “이렇게도 풀 수 있네”를 만들기

#### D. 희귀형 2개
실험성이 높거나,
비주류지만 기억에 남는 방향.

역할:
- IDEA MINE다운 발견감 제공
- 주말 특별 광맥, 희귀도 시스템과도 잘 연결됨

### 35-3. 어떤 축으로 다양화할 것인가
같은 광석 조합이라도 아래 해석 축을 바꾸면 서로 다른 원석이 나온다.

#### 1) 문제 해석 축
같은 타겟이라도 무엇을 문제로 보느냐가 달라질 수 있다.
예:
- 외로움 자체
- 생활 루틴 붕괴
- 자기 관리 부족
- 긴급 상황 대응 부족

#### 2) 사용 순간 축
언제 쓰는 서비스인지 바꿀 수 있다.
예:
- 아침 시작용
- 밤 시간대용
- 위기 순간용
- 주 1회 회고용

#### 3) 솔루션 형태 축
무슨 형태의 서비스인지 달라질 수 있다.
예:
- 챗봇형
- 루틴 코치형
- 에이전트 자동화형
- 커뮤니티 연결형
- 리포트/요약형

#### 4) 가치 실현 축
같은 Value라도 전달 방식이 다를 수 있다.
예:
- 직접 위로 제공
- 행동 유도
- 생활 편의성 개선
- 관계 연결

#### 5) BM 축
같은 Money라도 수익 방식의 결이 달라질 수 있다.
예:
- 개인 구독형
- 프리미엄 기능 잠금 해제
- 전문가 연결 수수료
- 기업 복지 B2B 확장

### 35-4. 추천 운영 규칙
실제 생성 단계에서는 아래 규칙을 두는 것이 좋다.

#### 규칙 1. 10개 중 최소 4개는 명확히 결이 달라야 한다
사용자가 “비슷한 말만 바꿨네”라고 느끼지 않아야 한다.

#### 규칙 2. 같은 문제 정의가 3개 이상 반복되면 안 된다
표현만 다른 유사 아이디어 군집을 막는다.

#### 규칙 3. 같은 제품 형태가 과반을 넘지 않게 한다
예를 들어 10개 중 7개가 모두 챗봇형이면 안 된다.

#### 규칙 4. 최소 2개는 의외성 있는 해석을 넣는다
안정형만 있으면 IDEA MINE 특유의 재미가 약해진다.

#### 규칙 5. 최소 2개는 실제 실행 가능성이 높은 방향을 유지한다
너무 실험적으로만 가면 선택 피로가 생긴다.

### 35-5. 다양화는 무엇을 기준으로 평가할까
좋은 다양화는 아래 4가지를 동시에 만족해야 한다.

- **관련성:** 광석 조합과 연결이 느껴진다.
- **차별성:** 서로 다른 결을 가진다.
- **선택 가능성:** 사용자가 “이 중 하나를 고를 수 있겠다”라고 느낀다.
- **기억성:** 최소 1~2개는 눈에 걸리는 원석이 있다.

### 35-6. 예시: 1인 가구 / 웰니스 / LLM 에이전트 / 외로움 해소 / 프리미엄
이 조합이라면 10개를 이런 결로 나눌 수 있다.

#### 안정형
- 감정 대화 + 루틴 관리
- 혼자 사는 사람용 저녁 체크인 에이전트
- 셀프 케어 플랜 제안형 서비스

#### 확장형
- 외로움 해소 대신 생활 관리 보조로 확장
- 대화형 위로보다 음성/일상 신호 기반 케어로 확장
- 개인용에서 반려동물/주거 환경 연계형으로 확장

#### 전환형
- 개인 구독형이 아니라 오피스텔/주거 서비스 제휴형
- 감정 케어보다 프리미엄 라이프 케어 컨시어지형

#### 희귀형
- 외로움 데이터를 시각화해 자기 이해를 돕는 감정 광산형 앱
- 특정 시간대의 고립감을 예측해 개입하는 선제형 에이전트

### 35-7. 추천 결론
다양화 규칙은 “많이 다른 것”보다
**의미 있게 다르게 해석된 후보군**을 만드는 쪽으로 가야 한다.

현재 IDEA MINE에는 아래 방식이 가장 적합하다.
- 10개를 안정형 / 확장형 / 전환형 / 희귀형으로 나눈다.
- 문제 정의, 사용 순간, 솔루션 형태, 가치 실현, BM 축을 섞어 다변화한다.
- 최소한의 실행 가능성과 최소한의 의외성을 동시에 유지한다.

## 36. 다양화 태그를 사용자에게 보여줄지 여부

현재 다양화 구조는 내부적으로 아래 4개 군을 사용한다.
- 안정형 3개
- 확장형 3개
- 전환형 2개
- 희귀형 2개

하지만 이 분류를 **사용자에게 그대로 노출할지**는 별도 판단이 필요하다.

### 36-1. 결론
초기 MVP에서는 이 4개 분류를 **내부 생성 로직용 태그로만 쓰고, 사용자에게는 그대로 보여주지 않는 것**이 가장 적절하다.

### 36-2. 그대로 노출했을 때의 장점
- 사용자가 원석들의 성격 차이를 빠르게 이해할 수 있다.
- 비교와 선택이 쉬워진다.
- 제품의 탐색 구조가 더 명시적으로 보인다.

### 36-3. 그대로 노출했을 때의 단점
- 시스템이 너무 공식처럼 보일 수 있다.
- 사용자가 라벨에 끌려 실제 내용보다 태그로만 고를 수 있다.
- `안정형/확장형/전환형/희귀형`이 브랜드 감성에는 맞지만, 실제 선택 UX에서는 다소 메타적으로 느껴질 수 있다.
- 생성 과정의 내부 분류가 전면에 드러나면 “발견의 재미”가 줄어들 수 있다.

### 36-4. 추천 방식: 내부 태그와 사용자용 표현을 분리한다
#### A. 내부 시스템 태그
모델 다양화와 중복 제어를 위한 로직용으로 사용.
- 안정형
- 확장형
- 전환형
- 희귀형

#### B. 사용자에게 보이는 표현
보다 가볍고 직관적인 “분위기/결” 수준의 태그만 노출.
예:
- `현실적`
- `트렌디`
- `의외성 있음`
- `실험적`
- `시장성 높음`
- `바로 써볼 만함`

이렇게 하면 내부 구조는 유지하면서도,
사용자 경험은 더 자연스럽게 만들 수 있다.

### 36-5. 단계별 추천
#### 홈 / 원석 선택 화면
- 내부 4분류는 숨긴다.
- 대신 필요하면 가벼운 분위기 태그만 노출한다.

#### Lab / 고급 탐색 모드
- 프리미엄 또는 고급 사용자용으로는
  원석을 `안정형 우선`, `실험형 더 보기` 같은 방식으로 필터링/정렬할 수 있게 확장 가능하다.

### 36-6. 최종 권장안
- **초기에는 안정형/확장형/전환형/희귀형을 사용자에게 직접 노출하지 않는다.**
- **대신 일부 원석에만 가벼운 분위기 태그를 붙여 선택을 돕는다.**
- **내부적으로는 4분류를 유지해 생성 품질과 다양성을 관리한다.**

## 37. 실험실(Lab) 확장 아이디어: 탐색 모드 필터

사용자 의견 기준으로, 내부 다양화 태그는 홈에서는 숨기고,
**실험실(Lab)에서만 탐색 옵션으로 노출**하는 방향이 적절하다.

### 37-1. 추천 방향
홈에서는 단순하고 직관적인 경험을 유지한다.
반면 실험실에서는 사용자가 좀 더 능동적으로 탐색 성향을 조절할 수 있다.

예:
- `안정형 더 보기`
- `실험형 위주로 보기`
- `시장성 높은 원석 우선`
- `의외성 있는 원석 더 보기`

### 37-2. 왜 실험실에 어울리는가
- 홈은 빠른 채굴용 공간이다.
- 실험실은 조합과 탐색을 더 깊게 만지는 공간이다.
- 따라서 내부 로직을 활용한 필터/탐색 옵션은 실험실에서 더 자연스럽다.

### 37-3. 제품 구조상 의미
이 방식은 아래 균형을 만든다.
- **홈:** 가볍고 게임적인 진입
- **실험실:** 더 정교한 탐색과 통제

즉, 초보자는 홈에서 쉽게 놀 수 있고,
파워 유저는 실험실에서 더 정밀하게 캘 수 있다.

## 38. 다음 논의 우선순위 업데이트

현재 흐름상 다음으로 상의할 항목은 아래 순서가 가장 자연스럽다.

### 1순위. 실험실(Lab)의 역할 정의
홈과 어떻게 다르고,
어디까지 직접 조절하게 할지 정해야 한다.

질문:
- 실험실은 단순 커스텀 조합 공간인가?
- 아니면 탐색 성향까지 제어하는 고급 모드인가?
- 무료와 유료의 경계는 여기서 어떻게 나뉘는가?

### 2순위. 원석 선택 이후 개요 문서로 넘어가는 흐름
사용자가 원석 10개 중 하나를 고른 뒤,
어떤 감정과 어떤 CTA로 개요 문서 생성까지 가게 할지 정해야 한다.

### 3순위. 개요 문서에서 유료 전환이 걸리는 지점
기본 개요는 높게 주되,
상세 MVP/Phase 설계는 어디서 열어줄지 구체화해야 한다.

### 4순위. 재방문/성장 장치의 상세 규칙
연속 접속 보상,
취향 광맥 해금,
주말 특별 광맥,
데일리 퀘스트를 어느 정도 깊이로 설계할지 정해야 한다.

### 5순위. 추천도 / 희귀도의 계산 의미
내부 로직 관점에서,
추천도와 희귀도가 어떤 기준으로 붙는지 정해야 한다.

## 39. 실험실(Lab) 역할 재정의

사용자 의견 기준으로, 실험실은 단순히 키워드를 더 만지는 공간이 아니라,
**모아둔 원석이나 아이디어를 실제 프로젝트 문서로 발전시키는 공간**으로 정의하는 것이 적절하다.

### 39-1. 핵심 정의
- **홈(The Mine)** = 새로운 가능성을 가볍게 채굴하는 곳
- **금고(The Vault)** = 원석과 개요서를 저장해 두는 곳
- **실험실(The Lab)** = 저장한 원석을 꺼내 실제 프로젝트 문서로 정제하는 곳

즉, 실험실은 “탐색”보다 한 단계 뒤의 공간,
곧 **정제와 발전의 공간**으로 보는 것이 맞다.

### 39-2. 왜 이 정의가 좋은가
이렇게 역할을 나누면 각 공간의 개성이 선명해진다.

#### 홈과의 차이
홈은 빠르고 가볍다.
실험실은 느리지만 깊다.

#### 금고와의 차이
금고는 저장 중심이다.
실험실은 저장된 것을 꺼내 가공하는 중심이다.

따라서 사용자는 흐름을 자연스럽게 이해할 수 있다.
- 홈에서 캔다
- 금고에 넣는다
- 실험실에서 다듬는다

### 39-3. 실험실의 핵심 역할
실험실에서 사용자가 할 수 있어야 하는 일은 아래와 같다.

1. 금고에 저장한 아이디어 원석 불러오기
2. 여러 원석을 비교하기
3. 하나를 선택해 프로젝트 개요 문서 생성하기
4. 개요 문서를 더 자세한 실행 문서로 확장하기
5. 필요하면 원석을 다시 조합하거나 변주하기

### 39-4. 실험실 안의 문서 단계
실험실은 하나의 문서만 만드는 곳이 아니라,
아이디어를 단계적으로 발전시키는 작업 공간으로 볼 수 있다.

#### 단계 1. 원석 선택
- 저장된 원석 중 하나를 고른다.

#### 단계 2. 개요 문서 생성
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM 초안
- MVP 범위

#### 단계 3. 확장 문서 생성 (유료/심화)
- 상세 MVP 기능 우선순위
- Phase 1 / 2 / 3 로드맵
- 사용자 플로우
- 실험/검증 포인트
- 성공 지표

이 구조는 무료/유료 구분과도 자연스럽게 연결된다.

### 39-5. 실험실의 제품 포지션
따라서 실험실은 “고급 탐색 모드”라기보다,
**아이디어를 실행 가능한 기획 문서로 바꾸는 제작실**에 가깝다.

필요하다면 내부에 보조 기능으로 아래를 둘 수 있다.
- 안정형 원석 더 보기
- 실험형 원석 위주로 보기
- 유사한 원석 다시 채굴
- 다른 BM으로 변주하기

하지만 본질은 어디까지나
**문서화와 정제**다.

### 39-6. 추천 결론
실험실은 아래처럼 정의하는 것이 가장 적합하다.

> 실험실은 금고에 저장된 아이디어 원석을 꺼내, 프로젝트 개요 문서와 실행 설계 문서로 발전시키는 정제 공간이다.

이 정의는
- 홈의 가벼운 게임성,
- 금고의 수집성,
- 실험실의 실무성
을 뚜렷하게 분리해 준다.

## 40. 실험실 문서화 흐름 고도화: 개요 문서 → 감정 → 정제

사용자 의견 기준으로, 실험실의 흐름은 단순히
`원석 선택 → 개요 문서 생성 → 끝`
이 아니라,
`원석 선택 → 개요 문서 생성 → 다양한 시각의 감정 → 정제/발전`
구조로 잡는 것이 더 적절하다.

### 40-1. 추천 흐름
#### 1단계. 원석 선택
금고에 저장된 아이디어 원석 중 하나를 고른다.

#### 2단계. 개요 문서 생성
선택한 원석을 바탕으로 기본 개요 문서를 만든다.
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM 초안
- MVP 범위

#### 3단계. 감정
생성된 개요 문서를 여러 관점에서 평가하고 해석한다.

예시 감정 축:
- 시장성 관점
- 사용자 문제 적합성 관점
- 실행 가능성 관점
- 차별화 관점
- 리스크 관점
- 확장성 관점

이 단계의 목적은 점수를 매기는 것보다,
**이 아이디어가 어떤 강점과 약점을 갖는지 입체적으로 보이게 하는 것**이다.

#### 4단계. 정제 / 발전
감정 결과를 바탕으로,
이 프로젝트를 어떤 방향으로 더 밀면 좋을지 조언하고,
필요하면 확장 문서를 만든다.

예:
- 문제 정의를 더 좁히기
- 타겟을 더 선명하게 바꾸기
- BM을 더 현실적으로 조정하기
- MVP를 더 작게 쪼개기
- 차별점을 더 강하게 살리는 방향 제안

### 40-2. 왜 이 구조가 좋은가
이 흐름은 단순 생성 툴보다 훨씬 강하다.

#### 장점 1. 사용자가 더 깊이 개입할 수 있다
아이디어를 그냥 받는 것이 아니라,
“검토하고 판단하고 개선하는 경험”으로 바뀐다.

#### 장점 2. 실무형 사용자에게 더 맞다
AI PM, 기획자, 창업 준비자는 아이디어 그 자체보다
“이게 왜 괜찮고, 어디가 약한지”를 더 궁금해한다.

#### 장점 3. 유료화 포인트가 자연스럽다
기본 개요는 무료로 주되,
감정 리포트의 깊이와 정제 단계의 실행 설계는 유료화하기 좋다.

### 40-3. 감정 단계의 올바른 역할
감정은 심사처럼 느껴지면 안 된다.
정답/오답을 판단하는 단계가 아니라,
**프로젝트를 입체적으로 바라보게 해주는 해석 단계**여야 한다.

따라서 감정은 아래처럼 작동하는 것이 좋다.
- 점수보다는 코멘트 중심
- 장점과 리스크를 같이 보여주기
- “이 아이디어는 별로다”보다
  “이 방향은 이런 상황에서 강하다 / 이런 점은 보완이 필요하다” 식으로 제시

### 40-4. 정제 단계의 올바른 역할
정제는 단순 재작성보다,
**프로젝트 방향을 더 선명하게 만드는 전략적 조언 단계**에 가깝다.

추천 역할:
- 무엇을 버릴지 제안
- 무엇을 더 강하게 밀지 제안
- 어디서부터 MVP를 시작할지 제안
- 누구를 첫 타겟으로 잡을지 제안
- 어떤 실험부터 할지 제안

즉, 정제는 “더 보기 좋게 쓰기”보다,
**더 실행 가능하게 방향을 잡아주기**에 초점을 둔다.

### 40-5. 추천 결론
현재 IDEA MINE에는 아래 흐름이 가장 적합하다.

> 원석 선택 → 개요 문서 생성 → 다양한 시각의 감정 → 방향 정제 및 실행 조언

이 구조는
- 게임적 발굴 경험,
- 기획적 검토 경험,
- 실무적 실행 연결
을 한 줄로 이어 준다.

## 42. 추천도 / 희귀도 정의

### 42-1. 한 줄 정의
- **추천도** = `지금 이 유저가 눌러볼 만한 정도`
- **희귀도** = `이 조합이나 해석이 얼마나 흔치 않은가`

즉,
추천도는 **적합성과 진입 장벽**에 가깝고,
희귀도는 **의외성과 발견감**에 가깝다.

### 42-2. 추천도는 무엇을 뜻하는가
추천도는 “좋은 아이디어”의 절대 평가가 아니다.
현재 시점에서 아래 요소를 종합해,
**사용자가 시작해 보기 좋은 광맥/원석인지**를 나타내는 신호다.

추천도에 반영할 수 있는 요소:
- 조합의 균형감
- 사용자 취향과의 적합도
- 이해하기 쉬운 문제 정의
- 초기 실행 가능성
- 최근 반응이 좋은 축과의 연결성

#### 쉽게 말하면
- `추천도 높음` = 지금 눌러도 실패 확률이 낮고, 시작하기 좋은 후보
- `추천도 보통` = 나쁘지 않지만 취향이나 상황을 탈 수 있는 후보
- `추천도 낮음` = 이해나 실행까지 한 단계 더 사고가 필요한 후보

### 42-3. 희귀도는 무엇을 뜻하는가
희귀도는 “좋다/나쁘다”가 아니라,
**얼마나 흔치 않은 조합/해석인지**를 뜻한다.

희귀도에 반영할 수 있는 요소:
- 흔히 나오지 않는 광석 조합
- 일반적이지 않은 문제 해석
- 서비스 형태의 의외성
- 실험성이 높은 방향
- 이벤트/특별 광맥 여부

#### 쉽게 말하면
- `희귀도 높음` = 자주 보지 못하는 결, 실험적이거나 눈에 띄는 후보
- `희귀도 보통` = 익숙하지만 약간 새로운 후보
- `희귀도 낮음` = 비교적 정석적이고 자주 이해되는 후보

### 42-4. 중요한 원칙
#### 1) 추천도와 희귀도는 서로 다른 축이다
- 추천도 높고 희귀도 낮을 수 있다.
  - 예: 현실적이고 이해하기 쉬운 안정형 조합
- 추천도 낮고 희귀도 높을 수 있다.
  - 예: 참신하지만 실행 방향이 아직 거친 실험형 조합
- 둘 다 높을 수도 있다.
  - 예: 드물지만 지금 사용자에게 매우 잘 맞는 발견형 조합

#### 2) 희귀하다고 더 좋은 것은 아니다
희귀도는 품질 점수가 아니라 **발견감 신호**다.

#### 3) 추천도가 높다고 덜 재미있는 것도 아니다
추천도는 사용자가 시작하기 쉬운 후보를 돕는 장치다.

### 42-5. 사용자에게 보이는 방식
초기에는 너무 숫자화하지 않는 것이 좋다.
배지나 짧은 표현 정도가 적절하다.

예:
- 추천도: `오늘의 추천`, `지금 파보기 좋음`, `입문용 광맥`
- 희귀도: `희귀 광맥`, `반짝 광맥`, `실험 광맥`

### 42-6. 제품 운영 관점의 권장 해석
#### 홈(광산)
- 추천도는 진입을 돕는 신호로 사용
- 희귀도는 발견의 재미를 주는 신호로 사용

#### 실험실
- 추천도 높은 원석 우선 보기
- 실험형 / 희귀 원석 위주로 보기 같은 탐색 기능으로 확장 가능

### 42-7. 추천 결론
현재 IDEA MINE에는 아래 정의가 가장 적합하다.

> 추천도는 “지금 이 유저가 시작해 보기 좋은 정도”,
> 희귀도는 “이 조합이나 해석이 얼마나 흔치 않고 의외성 있는가”를 뜻한다.

## 43. 금고(Vault) MVP 확정안

사용자 의견 기준으로, 금고는 기능이 많아 보이더라도 **새 메뉴를 추가해 분산시키기보다, 현재 구조 안에서 가벼운 기본 모드로 시작**하는 것이 적절하다.

### 43-1. 메뉴 구조 원칙
현재 단계에서는 금고 기능을 이유로 별도 메뉴를 늘리지 않는다.

이유:
- 홈 / 금고 / 실험실의 역할 분리가 이미 명확하다.
- 메뉴를 추가하면 제품이 무거워지고, 사용자가 어디서 무엇을 해야 하는지 헷갈릴 수 있다.
- IDEA MINE의 핵심인 “가볍게 들어와 채굴하고, 저장하고, 발전시키는 흐름”이 약해질 수 있다.

따라서 현재 구조는 아래처럼 유지한다.
- **홈** = 채굴
- **금고** = 저장 / 회수
- **실험실** = 발전 / 문서화

### 43-2. 금고 MVP의 핵심 역할
금고의 1차 역할은 아래 3가지다.
- 저장
- 회수
- 실험실 연결

즉, 금고는 모든 것을 복잡하게 관리하는 공간이 아니라,
**유망한 원석과 문서를 보관하고 다시 꺼내 다음 단계로 보내는 허브**로 본다.

### 43-3. 금고 MVP 저장 대상
초기 MVP에서 금고에 저장하는 대상은 아래 3가지로 제한한다.

1. **광맥 조합**
   - 다시 파보고 싶은 조합

2. **아이디어 원석**
   - 가능성 있는 raw seed

3. **프로젝트 개요 문서**
   - 실험실로 발전시키거나 다시 읽어볼 가치가 있는 문서

### 43-4. 금고 MVP에서 가능한 행동
초기 MVP에서는 아래 행동만 우선 제공한다.

#### 1) 다시 보기
- 저장한 광맥, 원석, 개요 문서를 다시 열람

#### 2) 즐겨찾기 / 핀 고정
- 중요한 항목을 위로 올려 정리

#### 3) 실험실로 보내기
- 원석 또는 개요 문서를 실험실에서 발전시키기

#### 4) 간단 메모
- 사용자가 짧은 판단을 남길 수 있음
- 예: “BM 약함”, “나중에 다시 보기”, “실버보다 1인 가구가 더 적합”

#### 5) 삭제 / 보관
- 금고를 쌓아두기만 하는 공간이 아니라 정리 가능한 공간으로 유지

### 43-5. MVP에서는 보류하는 기능
아래 기능은 가치가 있지만, 초기 MVP 범위에서는 제외한다.

- 원석/문서 비교
- 같은 원석에서 분기 만들기
- 감정 결과 상세 저장
- 취향 세트 관리
- 컬렉션/폴더 기능
- 공유/쇼케이스 보내기

이 기능들은 추후 금고 사용 패턴이 쌓인 뒤 확장한다.

### 43-6. 결론
금고의 MVP는 아래처럼 정의한다.

> 금고는 광맥, 아이디어 원석, 프로젝트 개요 문서를 저장하고,
> 사용자가 다시 보고, 간단히 정리하고, 실험실로 보내는 공간이다.

즉,
- 메뉴를 늘리지 않고,
- 기능을 과도하게 확장하지 않으며,
- 저장/회수/실험실 연결에 집중한다.

## 45. 재방문 시스템 상세 규칙 v1

사용자 의견과 피드백을 반영해, 재방문 시스템은 단순 보상 누적이 아니라
**채굴 활동이 쌓일수록 더 좋은 원석을 고르고, 더 나다운 광맥을 만나게 되는 성장 구조**로 정리한다.

### 45-1. 광부 레벨 시스템
광부 레벨은 단순 장식이 아니라,
사용자가 IDEA MINE 안에서 얼마나 꾸준히 채굴하고 발전시켰는지를 보여주는 성장 지표다.

#### 핵심 원칙
- 접속만으로는 소량의 경험치를 얻는다.
- 실제 행동에 더 많은 경험치를 부여한다.
- 즉, `들어오기`보다 `채굴하고 고르고 발전시키기`가 더 중요하다.

#### 경험치가 붙기 좋은 행동 예시
- 오늘의 광맥 1회 선택
- 아이디어 원석 1개 금고 저장
- 프로젝트 개요 문서 1회 생성
- 실험실에서 감정 완료

### 45-2. 레벨 보상: 금고 반입 슬롯
광부 레벨이 오를수록,
한 번의 채굴에서 금고로 가져갈 수 있는 원석 수가 늘어난다.

이 시스템은 “저장을 막는 제한”이 아니라,
**이번 채굴에서 어떤 원석을 회수할지 고르게 만드는 선택 장치**로 설계한다.

#### 추천 초기 구조
- 초기 레벨: 아이디어 원석 10개 중 2개까지 금고 반입 가능
- 이후 레벨 업: 3개 → 4개 → 5개까지 점진 확장

#### 설계 의도
- 금고가 무분별하게 쌓이는 것을 막는다.
- 사용자가 “무엇을 가져갈지” 판단하게 만든다.
- 성장할수록 더 많은 가능성을 회수할 수 있다는 보상감을 준다.

### 45-3. 데일리 퀘스트 구조
데일리 퀘스트는 재방문을 유도하는 보조 장치다.
핵심 채굴 루프보다 커지지 않게 유지한다.

#### 기본 데일리 퀘스트 후보
- 광맥 1회 선택하기
- 원석 1개 금고 저장하기
- 개요 문서 1회 생성하기
- 리롤 1회 사용하기

#### 보상 예시
- Pink Diamond 소량
- 추가 리롤 1회
- 금고 반입 보너스 1회권
- 희귀 광맥 확률 증가권

### 45-4. SNS 공유는 보너스 퀘스트로 처리
SNS 공유는 제품 확산에 도움이 될 수 있지만,
매일 강하게 요구하면 사용자 피로가 크다.

따라서 초기 MVP에서는 SNS 공유를 기본 데일리 퀘스트로 두지 않고,
**선택형 보너스 퀘스트**로 배치한다.

예:
- 마음에 드는 아이디어 공유하기
- 개요 문서 일부를 저장/공유하기

이 방식은 부담을 줄이면서도,
공유를 하고 싶은 사용자가 자연스럽게 참여하게 한다.

### 45-5. 주말 특별 광맥
주말 특별 광맥은 단순히 “AI 관련 단어 포함 확률 증가”로 설명하기보다,
**테마형 기술 광맥 이벤트**로 운영하는 것이 더 적절하다.

#### 운영 방향
- 주말에는 Tech 축이 강한 광맥 출현 확률을 높인다.
- 매주 같은 패턴이 아니라, 주말마다 테마를 달리한다.

#### 예시
- 에이전트 강화 주말
- 멀티모달 집중 주말
- AI x 헬스케어 광맥 주말
- AI x 생산성 광맥 주말

#### 설계 의도
- AI PM / 기획자 / 창업 준비자 타겟과 잘 맞는다.
- 주말마다 새로운 기대감을 만든다.
- 단순 확률 이벤트보다 브랜드 경험이 더 살아난다.

### 45-6. 취향 광맥 해금
취향 광맥은 꾸준히 들어온 사용자에게,
서비스가 “당신의 취향을 학습해 더 잘 맞는 광맥을 열어준다”는 감각을 주는 보상이다.

#### 핵심 원칙
- 단순한 재화 보상보다 IDEA MINE다운 보상으로 설계한다.
- 완전 만근형보다 약간 유연한 조건을 둔다.

#### 추천 조건 예시
- 주중 5일 중 4일 이상 활동 시
- 또는 최근 7일 중 5일 이상 채굴 시

#### 설계 의도
- 하루 놓쳤다고 전체 동기가 꺾이지 않게 한다.
- 꾸준한 사용이 실제로 더 좋은 경험으로 이어지게 한다.
- 취향 기반 탐색이라는 제품의 개성을 강화한다.

### 45-7. 재방문 시스템 요약
현재 IDEA MINE의 재방문 구조는 아래처럼 정리한다.

- **광부 레벨**: 활동 기반 성장 지표
- **금고 반입 슬롯**: 레벨 보상으로 확장되는 선택 권한
- **데일리 퀘스트**: 핵심 루프를 가볍게 한 번 더 돌리게 하는 장치
- **SNS 공유**: 보너스 퀘스트로만 운영
- **주말 특별 광맥**: 테마형 기술 광맥 이벤트
- **취향 광맥 해금**: 꾸준한 활동에 대한 개인화 보상

이 구조는
- 게임적 성장감,
- 저장과 선택의 긴장감,
- 반복 접속 동기,
- 제품 고유의 취향 학습 감각
을 함께 만든다.

## 46. 과금 구조 상세화: 무료 / 라이트 / 헤비

사용자 의견과 논의를 반영해, IDEA MINE의 과금 구조는 단순히 “많이 쓰는 사람 vs 적게 쓰는 사람”이 아니라,
**얼마나 깊게 아이디어를 발전시키는가**를 기준으로 나눈다.

즉,
- 무료는 가볍게 체험하고 한 번 더는 광고로 이어가는 구조,
- 라이트는 광고 없이 충분히 쓰는 실속형,
- 헤비는 실행 설계까지 깊게 쓰는 파워유저형으로 본다.

### 46-1. 과금 구조의 기본 원칙
#### 원칙 1. 처음 체험을 막지 않는다
아래 경험은 무료 사용자도 반드시 누릴 수 있어야 한다.
- 첫 채굴
- 첫 아이디어 원석 10개 확인
- 첫 프로젝트 개요 문서 생성
- 기본 감정

즉, 사용자가 서비스의 핵심 가치를 체감하기 전에 과금 장벽을 느끼게 하지 않는다.

#### 원칙 2. 유료는 “잠금 해제”보다 “더 깊게 파기”여야 한다
IDEA MINE의 유료화는 기능을 억지로 막아놓고 푸는 느낌보다,
**좋은 아이디어를 더 많이, 더 깊게, 더 실행 가능하게 만드는 권한**처럼 보여야 한다.

#### 원칙 3. 차이는 횟수보다 깊이에서 더 크게 난다
무료 / 라이트 / 헤비의 차이는 단순 사용량이 아니라,
- 문서의 깊이,
- 감정의 깊이,
- 실행 설계 가능 여부,
- 금고/실험실의 고급 기능 접근성
에서 분명히 드러나야 한다.

### 46-2. 1회 채굴 세션의 정의
과금과 사용량을 구분할 때의 기본 단위는 아래처럼 정의한다.

#### 1회 채굴 세션
- 광맥 1개 선택
- 아이디어 원석 10개 생성
- 그중 일부를 금고에 저장
- 원석 1개로 개요 문서 1개 생성

이 단위를 기준으로 무료, 광고 보상, 라이트, 헤비의 사용 범위를 설계한다.

## 47. 무료 + 광고형 구조

무료 유저는 IDEA MINE의 핵심 재미와 가치를 충분히 체험할 수 있어야 한다.
다만 반복 사용과 깊이 있는 활용은 광고 또는 구독으로 이어지게 한다.

### 47-1. 무료 기본 제공안
#### 하루 기준 권장안
- 랜덤 광맥 3개 제공
- 무료 리롤 2회
- 채굴 세션 1회
- 프로젝트 개요 문서 1회
- 기본 감정 1회

이 수준이면 무료 사용자도 “괜찮은 아이디어 하나를 캐고, 개요까지 보는 경험”을 할 수 있다.

### 47-2. 광고 보상형 확장
광고는 반드시 **보상형** 중심으로 운영한다.
강제 인터스티셜 광고는 몰입을 깨므로 지양한다.

#### 광고로 열어주기 좋은 항목
- 리롤 추가 5회
- 채굴 세션 1회 추가
- 프로젝트 개요 문서 1회 추가
- 기본 감정 1회 추가

즉, 무료 유저에게 광고는 “막힌 길”이 아니라
**한 번 더 이어서 채굴할 수 있는 추가 권한**으로 느껴져야 한다.

### 47-3. 광고를 넣지 말아야 할 순간
아래 구간은 사용 몰입도가 높은 구간이므로 광고 삽입을 피하는 것이 좋다.
- 원석 확인 중간
- 프로젝트 개요 문서 읽는 중간
- 감정 결과를 확인하는 중간

IDEA MINE은 문서와 아이디어 흐름이 중요하므로,
광고는 반드시 행동 전환 직후의 보상 지점에서만 사용한다.

## 48. 라이트 구독형 구조

라이트 구독형은 “가끔 진지하게 쓰는 사람”을 위한 플랜이다.
즉,
- 광고 없이 편하게 쓰고 싶고,
- 하루에 몇 번은 충분히 돌려보고 싶고,
- 개요 문서와 기본 감정을 꽤 넉넉하게 쓰고 싶은 유저에게 맞는다.

초기 타겟으로 보면,
- 부업형 기획자,
- 예비 창업자,
- 아이디어 수집형 유저,
- 가끔씩 본격적으로 쓰는 AI PM
에게 어울린다.

### 48-1. 라이트 플랜의 핵심 감정
라이트 플랜은 “부족한 플랜”이 아니라,
**생각날 때 꽤 본격적으로 써도 막히지 않는 실속형 플랜**처럼 느껴져야 한다.

### 48-2. 권장 제공 범위
#### 하루 기준 권장안
- 광고 제거
- 채굴 세션 5회
- 프로젝트 개요 문서 5회
- 기본 감정 5회
- 심화 감정 2~3회
- 실험실 확장 문서 1~2회

#### 추가 제공 가치
- 금고 저장 용량 확대
- 커스텀 광맥 / 직접 조합 제한적 허용
- 실험실 보조 기능 일부 개방

### 48-3. 라이트 플랜의 위치
라이트는 “무제한”이 핵심이 아니다.
핵심은 아래와 같다.
- 광고 없이 쾌적함
- 하루 안에서 꽤 넉넉한 시도 가능
- 개요 문서와 감정까지 충분히 사용 가능
- 실행 설계는 일부만 맛보기로 접근 가능

즉,
라이트는 **좋은 아이디어를 고르고 개요까지 만드는 사람**을 위한 플랜으로 정의한다.

### 48-4. 라이트 구독 외 보조 상품 가능성
라이트 유저는 매달 꾸준히 쓰는 사람만이 아니라,
특정 주간에 갑자기 많이 쓰는 사람일 수도 있다.

따라서 라이트 구조에는 추후 아래 같은 보조 상품을 붙일 수 있다.
- 개요 문서 추가 생성권
- 심화 감정 추가권
- 실험실 정제권

즉,
기본은 라이트 구독으로 두되,
필요할 때 소액 크레딧팩으로 보완하는 구조가 유연하다.

## 49. 헤비 구독형 구조

헤비 구독형은 IDEA MINE을 단순 아이디어 놀이가 아니라,
**실제 작업 도구**처럼 쓰는 파워 유저를 위한 플랜이다.

대상:
- AI PM
- 창업 준비자
- 반복적으로 아이디어 실험을 하는 기획자
- 프로젝트 문서를 실제로 발전시키는 사용자

### 49-1. 헤비 플랜의 핵심 감정
헤비 플랜은 “많이 쓸 수 있다”보다,
**아이디어를 실행 가능한 프로젝트로 밀어붙일 수 있다**는 감정이 중요하다.

### 49-2. 권장 제공 범위
#### 사용량 기준
- 광고 제거
- 채굴 세션 사실상 무제한 (내부 fair use 적용)
- 프로젝트 개요 문서 사실상 무제한
- 기본 감정 / 심화 감정 넉넉하게 제공
- 실행 설계 / MVP 설계 / phase 설계 반복 사용 가능

#### 고급 기능 기준
- 커스텀 조합 무제한
- 실험실 고급 모드
- 원석 비교 / 분기 / 재정제
- 금고 대용량 확장
- 취향 광맥 우선 해금
- 고급 테마 광맥 접근

### 49-3. 헤비 플랜의 위치
헤비는 단순히 라이트보다 횟수가 많은 플랜이 아니다.
헤비는 아래를 포함해야 한다.
- 더 깊은 감정
- 더 정교한 문서화
- 더 많은 재실험
- 실제 실행을 위한 설계 도구

즉,
헤비는 **개요 문서를 넘어, 아이디어를 실제 프로젝트로 내리는 사람**을 위한 플랜이다.

### 50-1. 무료

- 사실상 작업 도구 수준

### 50-4. 설계 원칙 요약
무료 / 라이트 / 헤비의 차이는 아래 비율로 보는 것이 적절하다.
- **횟수 40%**
- **문서 깊이와 작업 해상도 60%**

즉,
IDEA MINE의 유료화는 단순 사용량 차단보다,
**아이디어를 어디까지 발전시킬 수 있는가**의 차이를 중심으로 설계한다.

## 52. 유료 전환 순간과 문구 방향

사용자 의견과 논의를 반영해, IDEA MINE의 유료 전환은 “막힌 기능 해제”보다
**더 깊게 파기, 더 정밀하게 보기, 더 실행 가능하게 만들기**로 느껴져야 한다.

즉,
- 과금은 벌칙처럼 보이면 안 되고,
- 사용자가 이미 가치를 느낀 다음,
- “여기서 한 단계 더 가고 싶다”는 순간에 자연스럽게 열려야 한다.

### 52-1. 기본 원칙
#### 원칙 1. 첫 가치 체험 전에는 막지 않는다
아래 경험은 무료 유저도 충분히 할 수 있어야 한다.
- 첫 채굴
- 첫 아이디어 원석 10개 보기
- 첫 프로젝트 개요 문서 생성
- 첫 기본 감정

#### 원칙 2. 유료는 ‘잠금 해제’보다 ‘탐험 확장’처럼 보여야 한다
사용자에게는 아래 같은 감정이 들도록 설계한다.
- 더 깊게 파고 싶다
- 더 자세히 보고 싶다
- 이 아이디어를 진짜 프로젝트로 발전시키고 싶다

#### 원칙 3. 광고는 “한 번 더”, 구독은 “더 깊게”를 여는 구조로 나눈다
- **광고 보상형**: 리롤 추가, 채굴 1회 추가, 개요 문서 1회 추가, 기본 감정 1회 추가
- **구독형**: 정밀 감정, 실행 설계, MVP 청사진, 비교/분기/재정제, 광고 없는 플레이

## 53. 무료 한도 도달 시 문구 후보

### 53-1. 채굴 세션 소진
채굴 세션이 끝났을 때는 “막힘”이 아니라,
오늘 작업을 충분히 했고 더 하려면 추가 권한이 필요하다는 감정으로 보여준다.

#### 기본 문구 후보
- `오늘 채굴은 여기까지예요. 더 깊게 파려면 추가 채굴권을 열어보세요.`
- `오늘 광산 작업이 끝났어요. 야간 작업권으로 한 번 더 이어갈 수 있어요.`
- `오늘 채굴 가능한 광맥은 모두 확인했어요. 더 캐고 싶다면 광산주 권한이 필요해요.`

#### 광고 보상형 문구 후보
- `광고를 보고 야간 작업권을 받아 채굴을 1회 더 이어갈 수 있어요.`
- `전광판 광고 시청 후 추가 채굴 1회를 열 수 있어요.`

### 53-2. 개요 문서 생성 한도 도달
개요 문서는 아이디어를 정리된 결과물로 만드는 중요한 단계이므로,
“더 문서화하고 싶다”는 욕구에 맞춰 확장형 문구를 사용한다.

#### 문구 후보
- `오늘 만들 수 있는 개요 문서는 모두 완성했어요.`
- `이 원석을 문서로 더 발전시키려면 개요 확장권이 필요해요.`
- `기본 채굴은 끝났어요. 이제 더 많은 아이디어를 문서로 다듬으려면 Mine Owner 플랜이 어울려요.`

#### 광고 보상형 문구 후보
- `광고 시청 후 개요 문서 1회 추가 생성이 가능해요.`
- `이 원석을 놓치기 아쉽다면 정리권 1회를 열어보세요.`

### 53-3. 감정 심화 잠금
기본 감정 이후에는 “더 정밀하게 보고 싶다”는 니즈가 생기므로,
심화 감정은 유료 확장으로 자연스럽게 연결한다.

#### 문구 후보
- `기본 감정은 완료됐어요. 이제 정밀 감정으로 강점과 리스크를 더 깊게 볼 수 있어요.`
- `이 아이디어의 결을 더 자세히 보려면 심층 감정 리포트를 열어보세요.`
- `지금은 표면만 확인했어요. 더 깊은 해석은 광산주 전용 도구에서 제공돼요.`

### 53-4. 실행 설계 잠금
개요 문서에서 실제 프로젝트 실행으로 넘어가는 구간은 가장 강한 유료 전환 지점이다.

#### 문구 후보
- `개요는 완성됐어요. 이제 MVP 청사진으로 실제 실행 순서를 잡아볼 차례예요.`
- `이 아이디어를 프로젝트로 밀어붙이고 싶다면 실행 설계 확장이 필요해요.`
- `지금은 방향을 본 단계예요. 다음은 Phase 설계로 내려가는 단계예요.`

## 54. 라이트 → 헤비 업셀 방향

라이트에서 헤비로의 전환은 “더 많이 쓰기”보다,
**더 실행 가능한 단계로 이동하기**를 강조해야 한다.

### 54-1. 라이트의 역할
라이트는 아래 사용자에게 맞는다.
- 가끔 진지하게 쓰는 기획자
- 예비 창업자
- 아이디어 수집형 유저
- 광고 없이 충분히 쓰고 싶은 사용자

즉, 라이트는
**좋은 아이디어를 고르고 개요까지 만드는 플랜**이다.

### 54-2. 헤비의 역할
헤비는 아래 사용자에게 맞는다.
- 아이디어를 실제 프로젝트로 밀어붙이려는 사용자
- 반복 실험을 하는 AI PM / 기획자
- 실행 설계, 비교, 분기, 재정제를 자주 쓰는 사용자

즉, 헤비는
**아이디어를 실행 가능한 프로젝트로 만드는 플랜**이다.

### 54-3. 라이트 → 헤비 전환 문구 후보
- `아이디어를 모으는 단계는 충분히 하고 있어요. 이제 실행 가능한 설계 단계로 넘어가 볼까요?`
- `개요를 넘어 실제 프로젝트 구조까지 보고 싶다면 Pro가 더 잘 맞아요.`
- `지금부터는 좋은 아이디어보다 실행할 아이디어를 고를 단계예요.`
- `자주 실험하고 문서까지 발전시키고 있다면, 이제 광산주 Pro가 더 자연스러워요.`

### 54-4. 사용자 체감 차이 문장
- **Lite**: 아이디어를 고르고 정리하는 플랜
- **Pro**: 아이디어를 실행 가능한 프로젝트로 만드는 플랜

## 55. 유료 기능 이름 후보

유료 기능 이름은 제품의 세계관을 살리면서도,
실무적으로 무엇이 열리는지 명확히 느껴지게 해야 한다.

### 55-1. 감정 관련
#### 기본
- `감정`

#### 심화 / 유료 확장
- `정밀 감정`
- `심층 감정 리포트`
- `원석 감별 리포트`
- `가능성 진단`

현재 추천:
- 기본 단계 = `감정`
- 유료 심화 = `정밀 감정` / `심층 감정 리포트`

### 55-2. 실행 설계 관련
- `실행 설계`
- `MVP 청사진`
- `Phase 설계`
- `프로젝트 청사진`
- `실행 로드맵`
- `제품 설계 초안`

현재 추천 조합:
- 무료 = `프로젝트 개요`
- 유료 확장 = `MVP 청사진` / `실행 설계` / `Phase 로드맵`

### 55-3. 추가 사용권 관련
- `야간 작업권`
- `추가 채굴권`
- `정리권`
- `정밀 감정권`
- `실험실 패스`
- `개요 확장권`
- `청사진 확장권`

현재 추천 용도:
- 채굴 추가 = `야간 작업권`
- 개요 추가 = `정리권` 또는 `개요 확장권`
- 감정 심화 = `정밀 감정권`
- 실행 설계 = `실험실 패스` 또는 `청사진 확장권`

## 56. 티어별 감정 설계

### 56-1. 무료
무료는 아래처럼 느껴져야 한다.
- 하루에 한 번 들어와도 꽤 괜찮은 결과를 가져갈 수 있다.
- 마음에 들면 광고로 한 번 더 이어갈 수 있다.

핵심 감정:
- `가볍게 캐기`
- `한 번 더 해보기`

### 56-2. 라이트
라이트는 아래처럼 느껴져야 한다.
- 광고 없이 편하게 쓸 수 있다.
- 생각날 때 충분히 돌려볼 수 있다.
- 개요 문서와 기본 감정을 실속 있게 활용할 수 있다.

핵심 감정:
- `가볍지만 충분한 채굴 플랜`
- `생각날 때 막히지 않게`

### 56-3. 헤비
헤비는 아래처럼 느껴져야 한다.
- 이건 이제 놀이가 아니라 작업 도구다.
- 실제 프로젝트로 밀어붙일 수 있다.

핵심 감정:
- `실험실을 제대로 쓰는 플랜`
- `실행 가능한 수준까지 파고드는 플랜`

아래 3개를 중심으로 명확하게 잡는 것이 좋다.

### 57-1. 추가 개요 문서 생성
#### 문구
- `이 원석을 더 문서로 다듬고 싶다면 개요 확장권을 열어보세요.`

#### 문구

## 59. 무료 / Lite / Pro 기능 경계 확정안

현재 논의를 기준으로, 무료 / Lite / Pro의 경계는 아래처럼 고정한다.

### 59-1. 핵심 원칙
- **무료** = 가치 체험까지
- **Lite** = 반복 사용까지
- **Pro** = 실행 도구까지

즉,
무료도 충분히 좋아야 하고,
Lite는 광고 없이 여러 번 돌려볼 수 있어야 하며,
Pro는 실제 프로젝트 설계까지 가능해야 한다.

### 59-2. 무료 (Free Miner / 기본 광부)
#### 열어주는 범위
- 광산에서 오늘의 광맥 채굴
- 아이디어 원석 10개 보기
- 일부 원석 금고 반입
- 프로젝트 개요 1회 생성
- 기본 감정 보기
- 광고로 한 번 더 이어가기

#### 역할 정의
무료는
**원석 발굴 → 개요 생성 → 기본 판단**
까지 체험하는 단계다.

즉,
서비스의 재미와 실력을 모두 느끼게 하는 체험형 층이다.

### 59-3. Lite (Mine Owner Lite / 광산주 Lite)
#### 열어주는 범위
- 광고 제거
- 하루 여러 번 채굴
- 프로젝트 개요 여러 번 생성
- 기본 감정 넉넉하게 사용
- 정밀 감정 일부 사용
- 실험실에서 문서 발전 기능 일부 사용
- 금고 저장 여유 확대
- 커스텀 조합 일부 허용

#### 역할 정의
Lite는
**아이디어를 자주 캐고, 개요와 평가를 반복하는 실속형 플랜**이다.

즉,
정리와 평가까지 충분히 하는 사람에게 맞는다.

### 59-4. Pro (Mine Owner Pro / 광산주 Pro)
#### 열어주는 범위
- 채굴 / 프로젝트 개요 사실상 제한 없이 사용
- 정밀 감정 충분히 사용
- 실행 설계
- MVP 청사진
- Phase 로드맵
- 원석 비교 / 분기 / 재정제
- 고급 실험실 기능
- 큰 금고 / 정리 기능
- 커스텀 조합 완전 개방

#### 역할 정의
Pro는
**아이디어를 실제 프로젝트로 만드는 작업형 플랜**이다.

즉,
평가를 넘어 실행 설계, 우선순위, 로드맵까지 다루는 사람을 위한 단계다.

### 59-5. 체감 차이 요약
- **Free**: 채굴하고, 원석 보고, 개요 만들고, 기본 감정까지 해보는 체험형
- **Lite**: 광고 없이 여러 번 채굴하고, 개요와 평가를 반복하는 실속형
- **Pro**: 평가를 넘어 실행 설계, 우선순위, 로드맵까지 만드는 작업형

### 59-6. 설계 원칙 요약
무료 / Lite / Pro의 차이는 아래 비율로 본다.
- **횟수 40%**
- **문서 깊이와 작업 해상도 60%**

즉,
IDEA MINE의 과금 구조는 단순 사용량 제한보다,
**아이디어를 어디까지 발전시킬 수 있는가**의 차이를 중심으로 설계한다.

## 60. 실험실 확장 문서 체계 v1

현재 기준에서 실험실은 여러 개의 문서를 따로따로 만드는 공간이라기보다,
**하나의 프로젝트를 단계적으로 정제하며 완성도를 높여가는 공간**으로 정의한다.

즉, 사용자 경험은 아래 흐름으로 이어져야 한다.
- 프로젝트 개요
- 감정
- 방향 정리
- MVP 청사진
- Phase 로드맵

이 구조의 핵심은,
처음에는 가볍게 이해하고,
이후 점점 더 깊고 실행 가능한 설계로 내려가게 만드는 것이다.

### 60-1. 기본 원칙
#### 원칙 1. 사용자에게는 하나의 성장하는 프로젝트처럼 보이게 한다
내부적으로는 여러 문서 구조를 가질 수 있지만,
사용자에게는 `문서 4~5개를 따로 만드는 경험`보다
**하나의 프로젝트가 단계적으로 해금되고 깊어지는 경험**으로 보이는 것이 좋다.

#### 원칙 2. 앞 단계는 빠르게 판단하게 하고, 뒤 단계로 갈수록 실행 해상도를 높인다
- 앞 단계 = 가볍고 넓게
- 뒤 단계 = 좁고 깊게

#### 원칙 3. 무료 / Lite / Pro 차이는 횟수보다 단계 깊이에서 분명해져야 한다
- 무료 = 프로젝트 개요 + 기본 감정 + 짧은 방향 조언
- Lite = 반복 사용 + 정밀 감정 일부 + MVP 청사진 맛보기
- Pro = 실행 설계 전체 + MVP 청사진 전체 + Phase 로드맵 + 비교/분기/재정제

### 60-2. 단계별 구조
#### 1단계. 프로젝트 개요
원석을 바탕으로 이 아이디어가 어떤 프로젝트가 될 수 있는지 빠르게 이해하게 해주는 기본 문서.

#### 2단계. 감정
프로젝트 개요를 여러 관점으로 읽어주며,
이 프로젝트의 강점과 리스크를 입체적으로 해석하는 단계.

#### 3단계. 방향 정리
감정 결과를 바탕으로,
이 프로젝트를 어떤 방향으로 더 밀면 좋을지 전략적으로 조언하는 단계.

#### 4단계. MVP 청사진
실제로 첫 버전을 만든다면 어디까지를 MVP로 볼지,
핵심 기능과 검증 포인트를 구체화하는 단계.

#### 5단계. Phase 로드맵
프로젝트를 1단계, 2단계, 3단계로 어떻게 키워나갈지,
확장 순서를 보여주는 실행 흐름 단계.

### 60-3. 추천 결론
실험실은 아래처럼 정의하는 것이 가장 적절하다.

> 실험실은 저장한 아이디어 원석을 프로젝트 개요로 만들고,
> 감정과 방향 정리를 거쳐,
> MVP 청사진과 Phase 로드맵까지 발전시키는 단계적 정제 공간이다.

## 61. 실험실 단계별 정보 밀도 기준

실험실 단계별 정보는 모두 같은 밀도로 주면 안 된다.
핵심 원칙은 아래와 같다.

> 앞 단계는 빠르게 이해하고,
> 뒤 단계로 갈수록 더 실행 가능한 수준으로 깊어진다.

### 61-1. 프로젝트 개요
#### 역할
이 원석이 어떤 프로젝트가 될 수 있는지 빠르게 이해하게 하는 단계.

#### 적정 밀도
- 각 섹션당 3~5줄
- 전체 읽는 시간 1~2분 수준

#### 구성
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM 초안
- MVP 범위

#### 의도
“괜찮은 기획 초안”처럼 느껴질 정도로 충분히 좋아야 하지만,
감정이나 MVP 청사진 단계의 의미를 침범할 정도로 길어지지 않게 한다.

### 61-2. 감정
#### 역할
프로젝트를 여러 관점으로 읽어주는 해석 단계.

#### 적정 밀도
- 관점당 2~4문장
- 총 4~6개 감정 축
- 전체 읽는 시간 1분 내외

#### 추천 감정 축
- 시장성
- 문제 적합성
- 실행 가능성
- 차별화 가능성
- 확장성
- 리스크

#### 의도
감정은 점수표가 아니라,
**짧고 날카로운 해석 코멘트**로 작동해야 한다.

### 61-3. 방향 정리
#### 역할
그래서 이 프로젝트를 어떤 방향으로 밀어야 할지 정리해 주는 단계.

#### 적정 밀도
- 항목당 2~3줄
- 총 4~6개 제안
- 전체 읽는 시간 1~2분 수준

#### 추천 구조
- 더 좁혀야 할 타겟
- 더 강조해야 할 가치
- 버려도 되는 요소
- 먼저 검증해야 할 가설
- BM 조정 제안
- MVP 축소 제안

#### 의도
방향 정리는 “비평”보다
**판단을 돕는 전략적 조언**처럼 느껴져야 한다.

### 61-4. MVP 청사진
#### 역할
실제로 첫 버전을 만든다면 무엇부터, 어디까지 만들지 설계하는 단계.

#### 적정 밀도
- 각 섹션당 4~8줄
- 기능 우선순위는 리스트형 사용 가능
- 전체 읽는 시간 3~5분 수준

#### 추천 구조
- MVP 목표
- 핵심 타겟
- Must / Should / Later 기능 구분
- 첫 검증 포인트
- 추천 사용자 플로우
- 성공 기준 초안

#### 의도
MVP 청사진은 유료 가치의 핵심이므로,
사용자가 “이건 실제로 일에 쓸 수 있다”고 느낄 정도로 충분히 실무적이어야 한다.

### 61-5. Phase 로드맵
#### 역할
프로젝트를 단계적으로 어떻게 키워나갈지 보여주는 실행 흐름 단계.

#### 적정 밀도
- 각 Phase당 3~6줄
- 총 3개 Phase 정도
- 전체 읽는 시간 2~4분 수준

#### 추천 구조
- Phase 1 = 가장 작은 검증 버전
- Phase 2 = 리텐션 / 유료화 강화
- Phase 3 = 확장 / 차별화 / 스케일

#### 의도
로드맵은 개발 일정표가 아니라,
**의사결정의 흐름이 보이는 전략 문서**처럼 보여야 한다.

### 61-6. 티어별 정보 밀도 차이
#### 무료
- 프로젝트 개요 = 충분히 좋게
- 감정 = 기본 축소판
- 방향 정리 = 짧은 조언 수준

#### Lite
- 프로젝트 개요 = full
- 감정 = 기본 full
- 정밀 감정 = 일부
- 방향 정리 = full
- MVP 청사진 = 미리보기 수준

#### Pro
- 전 단계 full
- 정밀 감정 full
- MVP 청사진 full
- Phase 로드맵 full
- 비교 / 분기 / 재정제 포함

## 62. 실험실 단계별 예시 출력 샘플

아래 예시는 하나의 아이디어 원석이 실험실 안에서 어떤 방식으로 단계적으로 발전하는지 보여주기 위한 샘플이다.

### 62-1. 예시 아이디어 원석
- **원석 이름:** 밤 체크인 루틴 메이트
- **핵심 아이디어 한 줄:** 혼자 사는 사람이 외로움을 크게 느끼는 저녁 시간대에, AI가 짧은 대화와 생활 루틴 체크를 함께 제공하는 케어 서비스
- **보조 설명:** 감정 케어와 생활 습관 관리를 묶어서 재방문성과 지속 사용성을 높이는 방향의 원석

### 62-2. 프로젝트 개요 샘플
#### 문제 정의
혼자 사는 사람은 특히 저녁 시간대에 외로움, 무기력, 생활 루틴 붕괴를 동시에 겪는 경우가 많다. 기존의 감정 케어 앱은 위로나 기록에 집중하고, 생활 관리 앱은 생산성에 집중하는 경우가 많아 실제 일상에서 지속적으로 쓰이기 어렵다. 이 프로젝트는 감정 상태와 생활 루틴을 하나의 흐름으로 묶어, 사용자가 혼자 있는 시간을 더 안정적으로 보낼 수 있도록 돕는 것을 목표로 한다.

#### 타겟 유저
주요 타겟은 1인 가구 중에서도 저녁 시간의 고립감이나 생활 루틴 무너짐을 자주 경험하는 20~30대 직장인이다. 특히 퇴근 후 혼자 있는 시간이 길고, 감정 케어가 필요하지만 전문 상담 서비스까지는 원하지 않는 사용자를 1차 타겟으로 삼는다.

#### 핵심 기능
서비스는 사용자의 저녁 시간대에 맞춰 짧은 체크인 대화를 제공하고, 그날의 감정 상태와 생활 루틴을 함께 정리할 수 있게 한다. 사용자는 간단한 질문에 답하면서 감정 상태를 기록하고, 식사·샤워·수면 준비 같은 기본 루틴을 가볍게 점검할 수 있다. 이후 AI는 짧은 요약과 다음 행동 제안을 제공한다.

#### 차별점
이 서비스의 차별점은 감정 위로와 생활 루틴 관리를 분리하지 않고 하나의 흐름으로 묶는 데 있다. 단순 채팅형 케어보다 실사용성이 높고, 생산성 앱보다 정서적 진입장벽이 낮다. 즉, 감정 케어 앱도 아니고 할 일 앱도 아닌, 혼자 있는 시간을 관리하는 가벼운 저녁 동반자라는 포지션을 가진다.

#### BM 초안
기본 기능은 무료로 제공하고, 개인화된 루틴 제안, 주간 감정 리포트, 심화 체크인 모드 등을 프리미엄 구독 기능으로 제공할 수 있다. 장기적으로는 기업 복지, 주거 서비스, 생활 플랫폼과의 제휴 모델로 확장할 가능성도 있다.

#### MVP 범위
MVP에서는 저녁 시간 체크인 대화, 감정 상태 선택, 간단한 생활 루틴 체크, 하루 요약 생성 기능만 우선 검증한다. 복잡한 멀티모달 입력이나 전문 상담 연계는 제외하고, 사용자가 일주일 이상 반복해서 사용하는지부터 확인하는 것이 핵심이다.

### 62-3. 감정 샘플
#### 시장성
혼자 사는 사람의 외로움과 루틴 붕괴는 공감받기 쉬운 문제이기 때문에 초기 관심을 끌 가능성이 있다. 다만 위로만으로는 유료화가 어렵기 때문에, 생활 관리나 습관 유지와 연결되는 실용성이 반드시 함께 보여야 한다.

#### 문제 적합성
정서적 케어와 일상 루틴이 실제 사용자 문제 안에서 함께 나타난다는 점은 설득력이 있다. 특히 밤 시간대라는 특정 사용 맥락이 잡혀 있어서, 단순한 범용 챗봇보다 훨씬 선명한 문제 정의를 만들 수 있다.

#### 실행 가능성
초기 MVP는 텍스트 기반 대화와 간단한 체크인 구조만으로도 충분히 구현 가능하다. 다만 장기적으로 차별화를 유지하려면 대화 품질, 반복 사용성, 개인화 리포트 같은 요소가 중요해질 수 있다.

#### 차별화 가능성
감정 케어와 루틴 관리를 결합한 방향은 차별화 가능성이 있다. 하지만 유사한 웰니스 앱과의 경계가 흐려질 수 있으므로, 저녁 시간의 혼자 있는 순간이라는 맥락을 강하게 고정하는 것이 좋다.

#### 확장성
초기에는 개인용 서비스로 시작하되, 이후에는 주거 서비스, 직장인 웰니스, 기업 복지 등으로 확장 가능성이 있다. 감정 데이터와 루틴 데이터가 함께 쌓이면 개인화 기능의 확장 폭도 커진다.

#### 리스크
정서적 가치를 제공하는 서비스는 처음엔 호기심을 끌 수 있지만, 반복 사용 이유가 약하면 빠르게 이탈할 수 있다. 따라서 첫 버전부터 기분이 좋아지는 경험뿐 아니라 생활이 조금 정리되는 경험을 함께 줘야 한다.

### 62-4. 방향 정리 샘플
#### 1. 타겟을 더 좁히기
초기에는 1인 가구 전체보다 퇴근 후 혼자 있는 시간이 길고, 밤 시간대 루틴이 자주 무너지는 직장인으로 더 좁히는 편이 좋다. 타겟이 좁아질수록 메시지와 기능 우선순위가 선명해진다.

#### 2. 감성 가치보다 사용 습관 가치를 먼저 만들기
외로움 해소만 전면에 내세우면 추상적으로 보일 수 있다. 초기에는 밤 시간 루틴을 안정시키는 도구로 포지셔닝하고, 감정 케어는 그 안에서 자연스럽게 체감되게 하는 편이 더 강하다.

#### 3. 기능은 대화보다 체크인 흐름 중심으로
대화 자체를 길게 만드는 것보다, 짧은 체크인 → 감정 상태 선택 → 루틴 점검 → 요약 제안 흐름을 매끄럽게 만드는 것이 MVP에 더 적합하다. 사용자는 상담보다 가볍게 정리되는 느낌을 더 자주 원할 수 있다.

#### 4. BM은 프리미엄 구독보다 리텐션 검증이 먼저
처음부터 유료화보다, 사용자가 일주일 이상 반복 사용하는지부터 보는 게 중요하다. 이후 반복 사용이 확인되면 주간 요약, 개인화 루틴 추천, 심화 체크인 리포트를 프리미엄 기능으로 연결할 수 있다.

#### 5. 첫 검증 질문을 명확히 하기
가장 먼저 검증해야 하는 것은 사용자가 밤 시간에 이 서비스를 습관처럼 다시 여는가이다. 이 질문이 검증되지 않으면 감정 케어의 깊이나 고급 기능을 늘려도 유지율이 약할 수 있다.

### 62-5. MVP 청사진 샘플
#### MVP 목표
혼자 사는 사용자가 저녁 시간에 하루 1회 이상 서비스를 열고, 짧은 체크인을 반복 사용하는지 검증한다. 핵심은 감정 대화의 깊이보다, 체크인 루틴 자체가 재방문 행동으로 이어지는지를 보는 것이다.

#### 핵심 타겟
퇴근 후 집에 돌아와 혼자 시간을 보내는 20~30대 1인 가구 직장인. 특히 밤에 무기력하거나 생활 루틴이 흐트러지는 사용자를 핵심 대상으로 본다.

#### Must
- 밤 시간 체크인 시작하기
- 감정 상태 간단 선택
- 오늘의 생활 루틴 2~3개 체크
- 짧은 요약 메시지 제공
- 1일 기록 저장

#### Should
- 주간 요약 보기
- 체크인 시간 개인화
- 간단한 격려/루틴 제안
- 연속 체크인 표시

#### Later
- 음성 입력
- 멀티모달 감정 인식
- 전문가 연계
- 커뮤니티 기능
- 기업 복지 제휴형 기능

#### MVP 사용자 플로우
사용자는 밤 시간에 알림이나 자발적 진입으로 앱을 연다. 첫 화면에서 오늘 체크인을 시작하고, 현재 기분을 간단히 고른다. 이후 식사, 샤워, 수면 준비 같은 기본 루틴 몇 가지를 체크한 뒤, AI가 오늘 상태를 짧게 요약하고 한 가지 다음 행동을 제안한다.

#### 첫 검증 포인트
- 사용자가 일주일 안에 3회 이상 다시 들어오는가
- 체크인 완료율이 충분히 나오는가
- 감정 선택보다 루틴 체크가 더 자주 사용되는가
- 요약 메시지가 재방문 동기에 영향을 주는가

#### 성공 기준 초안
- 첫 주 재방문율 일정 수준 이상
- 체크인 완료율 일정 수준 이상
- 1회 사용 시간이 너무 길지 않고 부담이 적을 것
- 사용자 피드백에서 가볍게 정리된다는 반응이 확인될 것

### 62-6. Phase 로드맵 샘플
#### Phase 1 — 가장 작은 체크인 루프 검증
밤 시간 체크인, 감정 상태 선택, 생활 루틴 체크, 하루 요약 생성만 포함한 가장 작은 버전을 출시한다. 이 단계의 목표는 사용자가 반복적으로 돌아오는지, 그리고 대화보다 체크인 루프 자체가 유효한지를 검증하는 것이다.

#### Phase 2 — 개인화와 리텐션 강화
사용자의 반복 사용 데이터가 쌓이면 체크인 시간 개인화, 주간 요약, 맞춤형 루틴 제안 기능을 추가한다. 이 단계에서는 사용자가 단순 호기심을 넘어 습관적으로 서비스를 열게 만드는 것이 핵심이다. 프리미엄 구독 기능도 이 시점부터 연결할 수 있다.

#### Phase 3 — 확장과 수익화 다각화
개인용 서비스가 일정 수준 검증되면, 주거 플랫폼 제휴, 기업 복지형 패키지, 웰니스 제휴 모델 등을 검토한다. 동시에 음성 입력, 더 정교한 감정 분석, 장기 리포트 같은 확장 기능을 붙여 서비스의 차별화 폭을 넓힌다.

## 63. 실험실 단계 체감 요약

각 단계는 아래처럼 서로 다른 역할을 가진다.

- **프로젝트 개요** = 이게 무슨 프로젝트인지 빠르게 이해하는 단계
- **감정** = 이 프로젝트가 어디서 강하고 어디가 약한지 읽어주는 단계
- **방향 정리** = 그래서 어떤 방향으로 밀어야 할지 판단하게 해주는 단계
- **MVP 청사진** = 실제 첫 버전을 어떻게 만들지 잡아주는 단계
- **Phase 로드맵** = 그다음 어떻게 키울지 순서를 보여주는 단계

즉, 실험실의 핵심 리듬은 아래처럼 정리된다.

> 빠른 이해 → 입체적 판단 → 방향 선택 → MVP 설계 → 단계별 확장

## 64. 첫 사용자 온보딩 구조 v1

온보딩의 목표는 많은 기능을 설명하는 것이 아니라,
사용자가 IDEA MINE의 핵심 루프를 빠르게 이해하고 첫 채굴까지 자연스럽게 들어가게 만드는 것이다.

### 64-1. 온보딩에서 반드시 전달해야 할 3가지
1. 여기는 아이디어를 캐는 공간이다.
2. 채굴 결과는 아이디어 원석으로 나오고, 좋은 원석은 금고에 반입할 수 있다.
3. 원석은 실험실에서 프로젝트 개요와 더 깊은 문서로 발전시킬 수 있다.

즉,
온보딩은 설명 중심이 아니라
**짧은 소개 → 바로 첫 채굴 경험** 중심으로 설계한다.

### 64-2. 추천 첫 온보딩 흐름
#### 1단계. 짧은 세계관 소개
- 광맥을 고르면 아이디어 원석이 나온다.
- 좋은 원석은 금고에 넣을 수 있다.
- 원석은 실험실에서 프로젝트로 키울 수 있다.

#### 2단계. 오늘의 광맥 3개로 바로 진입
온보딩 직후 곧바로 오늘의 광맥 3개를 보여주고,
첫 행동은 반드시 `광맥 1개 선택`이 되도록 한다.

#### 3단계. 첫 아이디어 원석 10개 확인
첫 결과에서는 아래 핵심만 보여준다.
- 원석 이름
- 핵심 아이디어 한 줄
- 금고 반입
- 개요 만들기

#### 4단계. 금고 반입 1회 유도
사용자가 최소 한 번은 원석을 금고에 반입하게 하여,
서비스가 일회성 결과물이 아니라 자산이 쌓이는 공간이라는 감각을 만든다.

#### 5단계. 프로젝트 개요 1회 유도
저장한 원석 중 하나를 골라 프로젝트 개요까지 만들어 보게 하여,
IDEA MINE이 단순 아이디어 생성기가 아니라 프로젝트 발전 도구라는 점을 체감하게 한다.

### 64-3. 온보딩에서 과하게 설명하지 않을 것
처음부터 아래 내용을 길게 설명하지 않는다.
- Lite / Pro 차이
- 정밀 감정
- MVP 청사진
- Phase 로드맵
- 비교 / 분기 / 재정제
- 취향 광맥 해금 규칙

초기 온보딩에서는
`캐고 → 고르고 → 넣고 → 키운다`
이 흐름만 이해시키면 충분하다.

### 64-4. 온보딩 카피 방향
온보딩 문구는 설명서보다 초대처럼 느껴져야 한다.

예:
- 당신만의 광맥을 파보세요
- 좋은 원석은 금고에 넣어둘 수 있어요
- 마음에 드는 원석은 실험실에서 프로젝트로 키울 수 있어요

## 65. 온보딩 화면 1~5단계 문구 초안

### 65-1. 온보딩 1
#### 제목
아이디어는 찾는 게 아니라, 캐내는 거예요

## 67. Clean Mine Protocol 상세 운영안

IDEA MINE의 악성 유저 대응은 단순 금칙어 차단이 아니라,
**광산을 안전하게 유지하는 운영 체계**로 설계한다.

핵심 원칙은 아래와 같다.
- 말투는 부드럽게
- 기준은 단단하게
- 창피주기보다 수정 유도
- 반복 악용에는 명확한 제한

### 67-1. 역할 정의
Clean Mine Protocol은 아래 3가지 역할을 가진다.
1. 광산을 안전하게 유지한다.
2. 브랜드 톤을 해치지 않으면서 제지한다.
3. 반복 악용 시 실제 제한으로 연결한다.

### 67-2. 주요 차단 대상
초기 MVP 기준에서 강하게 관리하는 범주는 아래 4가지다.
- 음란/성적 표현
- 비속어/모욕
- 혐오/차별 표현
- 반복 스팸형 입력

### 67-3. 단계별 대응 구조
#### 1단계. 가벼운 오염 / 수정 유도
- 약한 비속어
- 애매한 표현
- 서비스 톤을 해치는 거친 입력

처리 원칙:
- 바로 차단하지 않고 수정 요청
- 대체 키워드 제안
- 채굴 시작 전 인라인 경고

대표 문구:
- `이 광석은 결이 조금 거칠어요. 다른 표현으로 다듬어볼까요?`
- `광산에서는 모두가 함께 쓸 수 있는 키워드만 채굴할 수 있어요.`
- `이 표현은 광산 규칙에 잘 맞지 않아요. 더 안전한 키워드로 바꿔보세요.`

#### 2단계. 명확한 오염 / 채굴 차단
- 명백한 음란 표현
- 강한 욕설
- 혐오/차별 표현
- 명확한 공격적 입력

처리 원칙:
- 해당 채굴 즉시 차단
- 생성 결과 없음
- 금고 / 실험실 진행 불가

대표 문구:
- `오염된 광석은 채굴할 수 없어요. 다른 조합으로 다시 시도해보세요.`
- `Clean Mine Protocol이 작동했어요. 안전한 광맥만 채굴할 수 있어요.`
- `이 입력은 광산 규칙에 맞지 않아 채굴이 중단됐어요.`

#### 3단계. 반복 오염 / 쿨다운
- 짧은 시간 안에 여러 번 차단
- 유사한 부적절 입력 반복
- 우회 시도 반복

처리 원칙:
- 일정 시간 채굴 제한
- 생성 / 리롤 / 문서화 일부 잠금
- 쿨다운 시간 명확히 표시

대표 문구:
- `반복된 오염 시도로 광산 환기 작업이 시작됐어요. 잠시 후 다시 시도해 주세요.`
- `안전 규칙을 여러 번 벗어난 입력이 감지되어 오늘 채굴이 잠시 제한됐어요.`
- `광산 안정화를 위해 잠시 작업이 중단됐어요.`

#### 4단계. 공개 악용 / 전시 차단
- 부적절한 프로젝트 공개 시도
- 쇼케이스 등록 기준 미달
- 신고 가능성이 높은 공개 콘텐츠

처리 원칙:
- 쇼케이스 등록 차단 또는 임시 숨김
- 실험실 재정제 유도
- 반복 시 공개 권한 제한

대표 문구:
- `이 프로젝트는 공개 광산 규칙에 맞지 않아 쇼케이스에 등록할 수 없어요.`
- `공개 전시를 위해서는 더 안전하고 명확한 프로젝트 설명이 필요해요.`
- `이 문서는 비공개로는 보관할 수 있지만, 현재 상태로는 전시할 수 없어요.`

### 67-4. 연출 원칙
브랜드 세계관은 유지하되 정책이 가벼워 보이지 않도록,
“실험실 폭발” 같은 코믹 연출보다 **오염 반응** 중심으로 표현한다.

추천 연출:
- 광석이 검게 변함
- 붉은 경고등 점등
- 작은 연기 / 노이즈 효과
- `오염 감지` 스탬프

### 67-5. 공개 기능과의 연결
나중에 쇼케이스/거래소가 열릴 경우,
Clean Mine Protocol은 아래와 연결된다.
- 쇼케이스 등록 전 최소 기준 확인
- 신고 접수 시 임시 숨김
- 반복 신고 / 위반 시 공개 권한 제한

## 68. 신뢰도 시스템 구조 v1

신뢰도 시스템은 단순 점수 경쟁이 아니라,
**성실한 활동을 보여주는 배지 체계 + 운영용 내부 신뢰 점수**로 구성한다.

### 68-1. 역할 정의
신뢰도 시스템의 목적은 아래와 같다.
1. 이 유저가 성실하게 활동하는 사람인지 보여준다.
2. 스팸형/악성 활동과 의미 있는 활동을 구분한다.
3. 좋은 활동을 보상하여 공개 공간의 품질을 높인다.

### 68-2. 3층 구조
#### 1층. 활동 배지
사용자가 프로필, 쇼케이스 등에서 보는 외부 신호

#### 2층. 신뢰 상태
배지보다 상위 개념의 상태 문구
예: 안정적 활동, 신뢰 가능한 기획자, 검증된 광산주

#### 3층. 내부 신뢰 점수
운영 로직 전용 지표
- 신고 이력
- 정책 위반 이력
- 공개 권한 판단
- 품질 판단 보조

### 68-3. 공식 배지 세트
#### 1) 꾸준한 광부
의미: 일정 기간 안정적으로 채굴한 유저

#### 2) 연속 채굴자
의미: 연속 채굴 streak 기반 활동 유저

#### 3) 정제 완료 기획자
의미: 원석을 프로젝트 개요까지 자주 발전시키는 유저

#### 4) 실험실 숙련자
의미: 감정, 방향 정리, 문서 심화까지 꾸준히 진행한 유저

#### 5) 신뢰 광산주
의미: 신고 없이 안정적이고 품질 있게 공개 활동하는 유저

#### 6) 전시 승인 기획자
의미: 쇼케이스 기준을 충족하는 프로젝트를 반복적으로 만든 유저

### 68-4. 배지 획득 조건 방향
#### 꾸준한 광부
- 최근 14일 중 7일 이상 활동
- 또는 누적 채굴 세션 일정 수 이상

#### 연속 채굴자
- 연속 채굴 3일 / 7일 / 14일 단계형

#### 정제 완료 기획자
- 프로젝트 개요 문서 일정 수 이상 생성
- 원석 → 개요 → 저장까지 이어진 활동 비중 반영

#### 실험실 숙련자
- 감정 완료 횟수 일정 기준 이상
- 방향 정리 또는 문서 심화까지 진행한 횟수 반영

#### 신뢰 광산주
- 일정 기간 신고 없음
- Clean Mine Protocol 반복 위반 없음
- 공개 활동 안정성 확보

#### 전시 승인 기획자
- 쇼케이스 등록 기준 충족 프로젝트 n개 이상
- 공개 후 신고/숨김 이력 없이 유지
- 감정/정제까지 거친 문서만 인정

### 68-5. 악용 방지 보정
배지 조건은 단순 생성량이 아니라 의미 있는 후속 행동에 가중치를 둔다.

원칙:
- 원석 대량 생성만으로는 배지 획득이 빠르지 않게 한다.
- 문서화 / 저장 / 감정 / 정제 같은 후속 행동 비중을 높인다.
- 반복 위반 시 배지 획득을 일정 기간 유예할 수 있다.
- 공개 계열 배지는 비공개 활동과 분리해 판단한다.

### 68-6. MVP 기준 권장 범위
#### 우선 노출 배지
- 꾸준한 광부
- 정제 완료 기획자
- 실험실 숙련자

#### 후속 확장 배지
- 연속 채굴자
- 신뢰 광산주
- 전시 승인 기획자

#### 본문 후보
- `안정적이고 믿을 수 있는 활동 이력이 쌓였어요.`
- `광산 규칙을 잘 지키며 프로젝트를 발전시킨 기록이 인정됐어요.`
- `이제 다른 사용자와 투자자에게도 더 신뢰되는 신호가 보여요.`

### 69-6. 전시 승인 기획자
#### 알림 제목
- `전시 승인 기획자 배지가 열렸어요`

#### 본문 후보
- `쇼케이스에 올릴 수 있는 수준까지 프로젝트를 다듬은 기록이 쌓였어요.`
- `공개 광산 기준을 안정적으로 통과하는 기획자라는 신호가 생겼어요.`
- `당신의 프로젝트는 이제 전시 가능한 완성도에 가까워지고 있어요.`

## 70. 프로필 / 쇼케이스 노출 방식 예시

배지는 보여주는 위치에 따라 역할이 달라져야 한다.

### 70-1. 프로필 노출 원칙
프로필에서는 대표 배지 1개를 크게, 보조 배지 1~2개를 작게 보여준다.
너무 많은 배지를 한 번에 노출하지 않는다.

#### 예시 카피 A
`도민 광부님`
`정제 완료 기획자 · 꾸준한 광부`

#### 예시 카피 B
`도민 광부님`
`실험실 숙련자`
`안정적 활동을 이어가는 기획자`

#### 예시 카피 C
`도민 광부님`
`신뢰 광산주 · 전시 승인 기획자`

### 70-2. 쇼케이스 노출 원칙
쇼케이스 카드에서는 최대 2개의 배지만 보여준다.
공개 공간에서는 신뢰와 품질 신호가 더 중요하므로,
`신뢰 광산주`, `전시 승인 기획자`, `실험실 숙련자` 계열을 우선 노출한다.

#### 예시 카피 A
`작성자: 도민`
`신뢰 광산주 · 실험실 숙련자`

#### 예시 카피 B
`도민`
`전시 승인 기획자`

#### 예시 카피 C
`도민`
`정제 완료 기획자 · 꾸준한 광부`

### 70-3. 거래소/투자자 공간 노출 원칙
거래소나 투자자 탐색 화면에서는 배지가 단순 꾸미기가 아니라
**신뢰 신호**로 읽혀야 한다.

추천 우선순위:
- 신뢰 광산주
- 전시 승인 기획자
- 실험실 숙련자

예시 카피:
- `검증된 광산주`
- `신뢰 가능한 기획자`
- `전시 기준 통과 이력 보유`

### 70-4. 금고 / 실험실 내부 노출 원칙
금고와 실험실에서는 배지가 메인 기능이 아니므로,
작고 가볍게만 노출한다.

예시 카피:
- `신뢰 이력이 쌓이고 있어요`
- `실험실 숙련자 배지 진행 중`
- `정제 완료 기획자까지 2개 문서 남음`

## 72. MVP / Phase 구조 정리

현재 기준에서 IDEA MINE의 제품 개발은 한 번에 모든 공간과 기능을 여는 방식보다,
**핵심 루프를 먼저 검증한 뒤 점진적으로 확장하는 Phase 구조**가 적절하다.

핵심 원칙은 아래와 같다.
- 먼저 “재미있고 다시 들어오고 싶은가”를 검증한다.
- 그다음 “계속 쓰게 만드는가”를 강화한다.
- 이후 “실제 기획/실행 도구로 쓸 수 있는가”를 고도화한다.
- 마지막으로 공개 생태계와 신뢰 시스템을 확장한다.

### 72-1. Phase 1 — 핵심 루프 MVP
#### 목표
광맥 선택 → 아이디어 원석 10개 → 금고 반입 → 실험실에서 프로젝트 개요 생성
이 핵심 루프가 실제로 매력적이고 재방문성을 만드는지 검증한다.

#### 핵심 질문
- 첫 채굴이 재미있는가?
- 원석을 금고에 반입하고 싶어지는가?
- 프로젝트 개요까지 가는 흐름이 자연스러운가?
- 다음날 다시 들어오고 싶어지는가?

### 72-2. Phase 2 — 리텐션 + 과금 강화
#### 목표
반복 사용 이유와 유료 전환 이유를 선명하게 만든다.

#### 확장 방향
- 정밀 감정
- Lite / Pro 차이 명확화
- 주말 특별 광맥 고도화
- 취향 광맥 해금
- 금고 저장 확장
- 실험실 방향 정리 강화
- MVP 청사진 미리보기 / 일부 개방
- 배지 시스템 초기 노출

### 72-3. Phase 3 — 실험실 고도화 + Pro 가치 강화
#### 목표
IDEA MINE을 실제 기획 / 실행 도구처럼 느끼게 만든다.

#### 확장 방향
- MVP 청사진 full
- Phase 로드맵 full
- 실험실 고급 기능
- 원석 비교
- 원석 분기
- 재정제
- 커스텀 조합 강화
- Pro 업셀 구조 정교화

### 72-4. Phase 4 — 공개 생태계 + 신뢰 시스템 확장
#### 목표
좋은 아이디어를 전시하고, 신뢰 가능한 기획자를 드러내고, 연결을 만든다.

#### 확장 방향
- 쇼케이스 오픈
- 공개 / 비공개 전환
- 신고 시스템
- 신뢰 광산주 / 전시 승인 기획자
- 거래소
- 투자자 탐색
- 공개 기준 검수
- 공개 공간용 배지 노출
- 전망대 일부 오픈

## 74. 메타 게임 확장 방향 (상위 개념만 반영)

IDEA MINE은 핵심적으로 아이디어 채굴과 문서화 도구이지만,
장기적으로는 **가벼운 게임 메타 루프**를 붙여 재방문성과 애착을 강화할 수 있다.

### 74-1. 기본 방향
메타 게임 요소는 본체를 대체하는 것이 아니라,
**채굴 활동에 보상을 붙여 사용자의 공간과 캐릭터가 함께 자라는 보조 루프**로 설계한다.

즉,
- 본체 루프 = 광맥 → 아이디어 원석 → 금고 → 실험실
- 메타 루프 = 활동 보상 → 공간/캐릭터 성장

### 74-2. 확장 가능한 메타 요소
- 광산 / 방 / 실험실 꾸미기
- 광부 캐릭터 꾸미기
- 가벼운 케어 요소
- 시즌성 / 희귀 보상형 아이템

### 74-3. 제품 원칙
- 메타 루프는 아이디어 기획 흐름을 방해하지 않는다.
- 꾸미기와 성장 요소는 숙제가 아니라 보너스 감정 루프로 작동해야 한다.
- 패널티형 관리보다 애착과 수집 욕구를 자극하는 방향이 적절하다.

### 74-4. Phase 관점 적용
- **Phase 1**: 본격 오픈하지 않거나 티저 수준으로만 예고
- **Phase 2 이후**: 기본 꾸미기와 캐릭터 커스터마이징 검토
- **후속 Phase**: 희귀 보상형 메타 루프와 가벼운 케어 확장 가능

### 74-5. 문서 관리 원칙
재화 구조, 꾸미기 상점, 희귀 보상, 밸런스 수치 등
세부 경제 설계는 난이도가 높기 때문에,
메인 제품 문서가 아니라 **별도 게임 경제/메타 루프 설계 문서**로 분리 관리한다.
