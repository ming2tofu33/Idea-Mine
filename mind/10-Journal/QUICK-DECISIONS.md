### 로드맵 Gate-Based 전면 개편

- **결정:** 기존 일직선 Sprint 계획(S1~S18)을 Improvement-Roadmap의 Gate-Based 접근과 머지. Phase 0(검증) -> Phase 1(3 Sprint Lean MVP) -> Gate 1 -> Phase 1.5(리텐션+과금) -> Gate 2 -> Phase 2(풀 픽셀+출시) 구조.
- **이유:** (1) Business-Audit에서 GTM 부재, 과잉 게임 경제, 검증 없는 개발 위험 지적 (2) Improvement-Roadmap이 Gate-Based 진행 제안 (3) 기존 계획은 Phase 1에 7 Sprint — 검증 없이 너무 많이 만듦 (4) 픽셀 아트 풀 투자는 Gate 통과 후로 미뤄야 낭비 방지
- **핵심 변경:** i18n 후순위(한국어 먼저), 픽셀 아트 최소 -> Gate 후 풀, Phase 0(2주 검증) 추가, GTM 병행, 48시간 throwaway 대신 실제 앱으로 프로토타입

### 아이디어 10개 4군 구조

- **결정:** 아이디어를 5개로 줄이는 대신, 10개를 유지하되 4개 군(안정3 + 확장3 + 전환2 + 희귀2)으로 구조화. 군 라벨은 사용자에게 노출하지 않고, 배치 순서만 안정->희귀 흐름으로.
- **이유:** Business-Audit의 "10개가 비슷한 말만 바꾼 거" 우려를 양 축소(5개)가 아닌 구조적 다양성 강제로 해결. 운영 규칙 5개를 프롬프트에 삽입해 품질 보장.
- **대안 기각:** 5개 축소(발상 폭 감소), 군 라벨 노출(과도한 정보), 랜덤 순서(첫 인상 불안정)
- **희귀도 시각/보상:** Phase 2에서 게임 경제와 함께 설계. Phase 1에서는 프롬프트 구조만.

### 티어 구조: Free/Lite/Pro 3단계 + AI 슬롯 차별화

- **결정:** Free Miner / Mine Owner Lite / Mine Owner Pro 3티어. AI 키워드는 Free=잠금, Lite=랜덤 배정, Pro=자유 선택.
- **이유:** 체험(Free) → 반복(Lite) → 실행(Pro) 경계가 선명. Lite도 6원석 품질을 경험하되, AI 기술 "선택권"은 Pro 전용으로 전환 동기 유지.
- **대안 기각:** Free/Pro 2티어(Lite 없음 — 반복 사용자 놓침), Lite도 AI 자유선택(Pro와 차이 약해짐)

### 커스텀 키워드 입력: Lite부터 개방

- **결정:** Lite부터 사용자가 카테고리 슬롯 내에서 키워드를 직접 입력 가능. AI 보정(임베딩 우선 + GPT-4o-mini 폴백)으로 품질 관리.
- **이유:** 반복 사용하는 Lite 유저가 자기 관심 분야에 맞는 키워드를 쓸 수 있어야 실속형 플랜의 가치가 체감됨. AI 보정 비용은 월 $3-24 수준으로 무시 가능.
- **대안 기각:** 완전 자유 입력(6슬롯 구조 흔들림), 보너스 슬롯 추가(구조 복잡), Pro 전용(Lite 가치 약해짐)
- **Lite/Pro 차이:** 입력 카테고리(5개/6개), 저장 수(10개/무제한), AI 보정 수준(기본/동의어+조합 추천)

### 아이디어당 키워드 부분 사용 (3~5개) + 풀 조합 Pro 전용

- **결정:** 각 아이디어는 광맥의 5~6개 키워드 중 3~5개만 선택해서 생성. "모든 키워드 사용(풀 조합)" 모드는 Pro 전용 옵션.
- **이유:** (1) 3~5개 부분 사용이 4군 구조의 다양성을 자연스럽게 만듦 (2) 2개까지 떨어지면 조합의 제약이 약해져 ChatGPT와 차이 없어짐 (3) 풀 조합은 "기본 모드에서 방향 찾기 -> 정밀 채굴"의 2단계 흐름으로 Pro 가치 강화
- **AI 키워드 고정:** Lite/Pro의 유료 AI 키워드는 항상 포함 (빠지지 않음). 나머지 5개 중 2~4개가 선택됨.
- **사용자 표시:** 각 아이디어 카드에 사용된 키워드를 태그로 표시.

### AI 범용 키워드 교체

- **결정:** "생성형 AI(Generative AI)" -> "AI 에이전트 워크플로우(AI Agent Workflow)", "자연어 처리(NLP)" -> "실시간 번역 AI(Real-time Translation AI)"로 교체.
- **이유:** 생성형 AI와 NLP는 상위/기반 개념이라 6번째 슬롯에 넣어도 아이디어 방향이 안 바뀜. 유료 업셀 동기를 약화시킬 수 있음. 교체 키워드는 구체적이어서 5원석 vs 6원석 품질 차이가 체감됨.
- **대안 기각:** AI 요약/정리(retrieval subtype 이미 4개로 많음), 대화형 AI(챗봇 Tech 키워드와 겹침)

### 당연한 조합 회피 규칙 + 희귀형 가드레일

- **결정:** 광맥 생성 시 의미 겹침 회피 3규칙 추가. 희귀형은 키워드 3개만 사용하는 구조 덕에 별도 가드레일 최소화 — 운영 규칙 6번 "실제 사용자가 존재할 수 있는 범위" 추가로 충분.
- **이유:** (1) "시니어 x 헬스케어 x 건강 트래킹" 같은 당연한 조합은 세렌디피티 엔진 무력화 (2) 희귀형이 3개 키워드만 쓰면 자연히 황당해지지 않음 — 구조적 해결

### MCP 서버: 영구 무료 획득 채널

- **결정:** IDEA MINE MCP 서버를 영구 무료 획득 채널로 개발. 제품(앱)이 아닌 채널. Phase 1 Sprint 3 완료 후 1~2일에 배포.
- **이유:** (1) MCP 설치 = 마찰 제로 체험 (앱 설치 6단계 vs MCP 2단계) (2) AI 코딩 도구 사용자 = 핵심 타겟에 직접 닿음 (3) 결과 휘발성이 자연스러운 앱 전환 동기 (Loss Aversion) (4) 백엔드 API 위 얇은 래퍼라 1~2일 개발
- **기술:** Python + FastMCP. 백엔드 API 위 래퍼. 비즈니스 로직 0줄.
- **기능:** 앱 Free와 동일 (광맥 3개, 리롤 2회, 채굴 1회/10개, 개요서 1회). 저장 불가.
- **비용 트래킹:** API source 태그 (`app` | `web` | `mcp`)로 채널별 비용 분리 추적.
- **대안 기각:** MCP를 제품으로 만들기 (세계관/픽셀 아트 차별점 사라짐), Node.js MCP (백엔드가 Python인데 이중 관리)

### 아이디어 생성 파이프라인 v2: Python 키워드 선택 + LLM 생성 분리

- **결정:** LLM이 키워드 선택과 아이디어 생성을 동시에 하던 v1 구조를 폐기. Python이 4군별 키워드 조합 10세트를 미리 뽑고, LLM에게 "이 조합으로 아이디어 1개 만들어"를 10번 요청 (1회 API 호출).
- **이유:** (1) v1에서 LLM이 slug를 추측해서 keyword_combo 빈 배열 버그 발생 (2) "다른 조합 선택하라"는 추상적 지시로는 다양성 보장 불가 (3) 키워드 선택은 규칙 기반이므로 Python이 더 적합, LLM은 창의적 생성에만 집중
- **키워드 조합 방식:** 순수 랜덤. 안정형 4~5개, 확장형 3~4개, 전환형 3~4개, 희귀형 정확히 3개. AI 키워드는 Lite/Pro에서 고정 포함.
- **대안 기각:** 겹침 최소화 알고리즘(과도한 복잡성), 카테고리 기반 규칙(패턴 생겨서 뻔해짐)

### 프롬프트 영어 작성 + 한/영 동시 출력

- **결정:** 프롬프트는 영어로 작성하고, LLM 출력은 한국어/영어 둘 다 동시 생성.
- **이유:** (1) LLM이 영어 프롬프트를 더 잘 이해 (2) 한/영 동시 생성하면 DB에 양쪽 저장 → 유저가 언어 전환 시 재생성 비용 0 (3) 글로벌 서비스 대비
- **DB 변경:** ideas 테이블 `title` → `title_ko` + `title_en`, `summary` → `summary_ko` + `summary_en`, `language` 삭제

### Sprint 2 홈 화면 디자인 확정

- **결정:** Phase 1은 카드 UI(하이브리드), Gate 1 통과 후 픽셀 광산 씬으로 업그레이드.
- **이유:** 광산 씬 에셋 제작 비용이 큼. 핵심 루프 검증에 광산 씬은 필수 아님. Gate-Based 철학에 맞게 카드 UI로 빠르게 검증 → 증명되면 투자.
- **상태바:** 2줄 (프로필 + 자원). 닉네임, Lv, 연속접속, 채굴/리롤/가방/재화 표시.
- **닉네임:** 첫 진입 시 모달로 입력 (온보딩은 Sprint 7).
- **채굴 소진:** 광맥 유지 + "광고로 1회 추가" 자리 확보 (Phase 1.5에서 연동).
- **로딩 연출:** 세계관 단계별 텍스트 ("광맥을 스캔하는 중..." → "아이디어 결정을 추출하는 중...").

### 광맥 희귀도: 시즌/트렌드 기준 + 광산 메타포 네이밍

- **결정:** 광맥 희귀도를 시즌/트렌드 기반(C)으로 결정. 네이밍은 광산 메타포 사용: 일반 광맥(Common) → 금빛 광맥(Golden) → 전설 광맥(Legend).
- **이유:** (1) 시즌 기준 = 구현 단순 + FOMO 리텐션 + 소셜 콘텐츠 연결 (2) "전설 광맥"이 "한정 광맥"보다 감정적 무게가 크고 공유 욕구 유발 (3) "기본/특별/한정"은 마케팅 용어 느낌, "일반/금빛/전설"은 광산 세계관과 자연스러움
- **대안 기각:** 의미적 거리 기준(A — 임베딩 계산 필요, 설명 어려움), 확률 배분(B — 확률 고지 필요), 깊이 메타포(지표/심층/고대 — 시즌과 직접 연결 안 됨), 빈도 메타포(기본/특별/한정 — 특별함 부족)
- **등급별 정의:**
  - 일반 광맥: 기본 키워드 풀(109개)에서 조합. 상시 등장.
  - 금빛 광맥: 최근 AI 트렌드 키워드 1개 이상 포함. 트렌드 기간에 등장.
  - 전설 광맥: 시즌 한정 키워드 포함. 해당 시즌에만 등장, 시즌 종료 시 사라짐.
- **등급 배분:** 확률 기반. 비시즌에도 금빛 소량 등장. 확률 고지 감수.
- **확률표:**

  | 조건 | 일반 | 금빛 | 전설 |
  |------|------|------|------|
  | 비시즌 평일 | 97% | 3% | 0% |
  | 비시즌 주말 | 94% | 6% | 0% |
  | 시즌 평일 | 90% | 8% | 2% |
  | 시즌 주말 | 82% | 13% | 5% |

  *광맥 3개 각각에 독립 적용. Free 유저는 리롤 포함 최대 9개 광맥 → 비시즌 평일에도 금빛 24% 확률.*

- **금빛 vs 전설 키워드 차이:**
  - 금빛 키워드 = 넓은 트렌드 (업계 핫 키워드 3~5개, 분기 내내 유지)
  - 전설 키워드 = 시즌 테마에 맞춘 구체적 키워드 세트 (카테고리별 1개씩 5~6개, 시즌 이벤트 기간 ~1개월만)
- **확률 고지:** 앱 내 광맥 화면 물음표(?) 아이콘 + 설정 > 확률 정보 + 앱스토어 설명에 명시. 한국 게임산업진흥법 + Apple/Google 정책 준수.
- **색상:** 일반=회청, 금빛=골드+goldGlow, 전설=보라/핑크+purpleGlow
- **시즌 키워드 범위:** 모든 카테고리에 시즌 키워드 추가 (AI뿐 아니라 Who, Domain, Tech, Value, Money에도). Free 유저도 전설 광맥을 만날 수 있음. 시즌 종료 시 해당 키워드 제거.
- **대안 기각:** AI 카테고리에만 추가(Free가 시즌 경험 불가 → 리텐션 약화), 별도 큐레이팅 광맥(운영 부담), 고정 배분(확률 고지 불필요하지만 짜릿함 없음)

### 가방 용량 Lv1=2개 유지 + 튜토리얼 즉시 레벨업

- **결정:** 가방 시작 용량 2개 유지. 단, 튜토리얼 완료 시 즉시 Lv2 → 가방 3개로 확장. "박함"이 아니라 "첫 성장의 쾌감"으로 전환.
- **이유:** (1) 2개는 박하지만, 5분 내 3개로 올라가면 체감이 다름 (2) "처음엔 2개였는데 금방 3개 됐네" = 성장 보상감 (3) 게임 튜토리얼 보상 패턴과 동일 (4) 가방 용량 테이블(Lv1=2, Lv2=2→3 등) 자체는 변경 불필요, 튜토리얼 완료 시 레벨업 로직만 추가
- **대안 기각:** Lv1을 3개로 변경(성장감 없음), Lv1을 1개로 변경(너무 박함)
- **구현 시점:** Phase 1.5 (온보딩/튜토리얼과 함께)

### 리롤 = 전부 새로 뽑기 (채굴된 아이디어는 금고에 보존)

- **결정:** 리롤 시 오늘 광맥 3개를 전부 새로 뽑음. 이미 채굴해서 생성된 아이디어/개요서는 금고에 그대로 남아있음. 광맥은 "오늘의 선택지"일 뿐 일회용.
- **이유:** (1) 광맥에 미련 남아도, 이미 채굴했으면 원석이 금고에 있으니 괜찮음 (2) 채굴 안 했는데 리롤하면 유저의 선택 (3) FK를 SET NULL로 변경하면 아이디어가 광맥 삭제 후에도 독립 존재 가능
- **나중에 추가:** "이 광맥 저장하기" (Phase 2 취향 광맥), 리롤 전 확인 모달

### 실험실 파이프라인 5단계 + 정보 밀도 설계

- **결정:** 실험실 흐름을 5단계(개요 → 감정 → 방향 정리 → MVP 청사진 → Phase 로드맵)로 확정. 각 단계의 정보 밀도와 티어별 깊이를 설계.
- **이유:** (1) 단계별로 역할이 명확히 다름 — 개요는 빠른 이해, 감정은 입체적 판단, 방향은 행동 조언, 청사진은 실무 설계, 로드맵은 전략 (2) 티어별 깊이 레이어로 "부족함"이 아닌 "더 깊게 보고 싶음"을 유발
- **상세:** `The-Lab` 참조

### 개요서-감정 완전 분리 + 개요서 프롬프트 재설계

- **결정:** (1) 개요서에서 점수/평가를 완전 제거하고 감정 단계로 이동 (2) 개요서 생성 모델을 GPT-4o-mini → GPT-4로 변경 (3) 개요서 구성: 문제 정의, 타겟 유저, 핵심 기능, 차별점, BM 초안, MVP 범위 (4) 밀도: 섹션당 3~5줄, 1~2분 읽기
- **이유:** (1) 개요서와 감정의 역할이 혼재되어 각 단계 가치가 약해짐 (2) 4o-mini는 분석 깊이가 얕아 개요서 품질이 부실 (3) 개요서는 제품 가치를 증명하는 핵심 순간이라 모델 품질이 중요
- **대안 기각:** 개요서에 미니 평가 유지(감정과 역할 겹침), 4o-mini 유지(품질 부족)

### 감정은 점수가 아니라 코멘트

- **결정:** 감정 단계의 출력을 숫자 점수(7/10)가 아닌 짧고 날카로운 코멘트로 변경. 축당 2~4문장, 4~6개 축.
- **이유:** (1) 점수는 "그래서 뭐?"로 끝남 — 행동으로 연결 안 됨 (2) 코멘트는 읽자마자 강점/약점이 파악되어 다음 단계(방향 정리)로 자연스럽게 이어짐 (3) "시장성 7점"보다 "수요 공감은 쉽지만 지불 의사로 연결되려면 기능 선명도가 더 필요함"이 훨씬 실용적
- **감정 축:** 시장성, 문제 적합성, 실행 가능성, 차별화 가능성, 확장성, 리스크
- **대안 기각:** 점수+코멘트 병행(점수가 있으면 코멘트를 안 읽음), 점수만(현재 문제의 원인)

### 실험실 파이프라인 정보 밀도 설계

- **결정:** 5단계 파이프라인 각 단계의 정보 밀도와 읽기 시간을 설계. 개요(1~2분, 기획초안) → 감정(1분, 날카로운 코멘트) → 방향 정리(1~2분, 코치 조언) → MVP 청사진(3~5분, 실무 설계) → Phase 로드맵(2~4분, 전략 문서).
- **이유:** 단계마다 역할이 다르므로 밀도도 달라야 함. 개요에서 너무 깊으면 감정의 의미가 약해지고, MVP 청사진이 얕으면 Pro 가치가 안 느껴짐.
- **밀도 원칙:** 개요는 짧고 선명 / 감정은 날카롭고 압축 / 방향은 행동 조언 / MVP는 실무적 충분 / 로드맵은 전략적 간결
- **상세:** `The-Lab` 참조

### 개요서 프롬프트 v3 재설계

- **결정:** 개요서 프롬프트를 v2→v3로 전면 재작성. (1) 점수/평가 섹션 완전 제거 (2) 차별점(differentiator), MVP 범위(mvp_scope) 섹션 신규 추가 (3) 모델 gpt-4o-mini→gpt-4o 변경 (4) 구체성 규칙 강화 ("Users aged 25-40" 금지, AI 클리셰 금지) (5) 톤 변경: "evaluate" → "describe"
- **이유:** v2 개요서가 부실했던 근본 원인: (1) 개요와 감정 혼재로 LLM 인지 부하 (2) MRR 계산 유도가 분석 대신 산수를 만듦 (3) 4o-mini 분석 깊이 한계 (4) 차별점/MVP 범위 부재로 "그래서 뭐가 다른데?"에 답 못함
- **영향 파일:** `overview.py`, `overview_service.py`, `overview.ts`, DB overviews 테이블 (마이그레이션은 Sprint 3에서)

### 감정 프롬프트 v1 신규 설계
