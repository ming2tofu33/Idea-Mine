# 2026-03-22 — MCP 채널 도입 결정

> MCP 서버를 왜 만들기로 했는지, 어떤 포지셔닝으로, 어떤 판단이 들어갔는지.

---

## 배경

GTM-Strategy를 설계하면서 "코드를 쓰는 기획자/개발자"가 IDEA MINE의 1차 타겟이라는 게 명확해졌다. 이 사람들은 이미 Claude Code, Cursor, Windsurf 같은 AI 코딩 도구를 매일 쓴다.

그런데 앱을 깔게 만드는 건 허들이 높다. 반면 MCP는 이미 쓰고 있는 도구 안에서 바로 체험할 수 있다. "앱 다운로드"가 아니라 "MCP 추가"면 마찰이 거의 없다.

---

## 핵심 판단: 제품이 아니라 획득 채널

MCP 서버를 만들 때 가장 중요한 결정은 **"이건 제품이 아니라 획득 채널이다"**였다.

이게 왜 중요하냐면:
- **제품으로 만들면:** MCP에서도 저장/관리/배지/구독을 구현해야 → 개발 비용 폭증
- **채널로 만들면:** 핵심 루프만 텍스트로 체험 → 결과는 휘발 → 앱으로 유도

앱의 Free 티어와 동일한 경험(광맥 3개, 리롤 2회, 채굴 1회, 개요서 1회)을 제공하되, **저장이 안 된다**는 게 핵심 전환 동기. 횟수 제한은 "짜증"을 주지만, 휘발성은 "아쉬움"을 준다. Loss Aversion 관점에서 아쉬움이 더 강한 전환 동기.

---

## 아키텍처 판단: MCP에 비즈니스 로직 0줄

MCP 서버는 **백엔드 API의 얇은 래퍼**로만 동작한다.

```
AI 어시스턴트 → MCP 서버 (래퍼) → 백엔드 API → OpenAI / Supabase
```

이렇게 한 이유:
1. **유지보수 단일화** — 프롬프트 수정, 키워드 추가, 4군 규칙 변경 → 백엔드만 고치면 앱+MCP 양쪽 반영
2. **빠른 개발** — Sprint 3 백엔드 API 완성 후 1~2일이면 MCP 래퍼 배포 가능
3. **비용 예측 가능** — 앱 Free와 동일한 비용 구조. 별도 모델 불필요

---

## 세계관 톤: 괄호 병기

MCP에서도 세계관 용어를 쓰되, 처음 접하는 사용자를 위해 괄호로 일반 용어를 병기하기로 했다.

예: "광맥(키워드 조합)", "원석(아이디어)", "금고(저장소)"

세계관 없이 "아이디어 생성기"라고만 하면 IDEA MINE의 정체성이 사라진다. 괄호 병기로 즉시 이해 + 앱 넘어갔을 때 자연스러운 연결.

---

## 비용 판단

| 규모 | 월 비용 |
|------|--------|
| 100 DAU | ~$5 |
| 1,000 DAU | ~$50 |
| 10,000 DAU | ~$500 |

앱 Free 티어와 동일한 비용이라 별도 수익 모델 필요 없음. 획득 비용 대비 극히 저렴한 채널.

---

## 타이밍

Sprint 3에서 백엔드 API(광맥 생성, 채굴, 개요서)가 완성되면 그 위에 MCP 래퍼를 씌운다. 앱 출시 전에 MCP를 먼저 배포하면 early adopter를 모을 수도 있다.
