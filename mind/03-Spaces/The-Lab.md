# The Lab (실험실 / 믹스)

> 직접 조합하고 실험하는 공간. 무료보다 유료 플랜의 가치를 강하게 체감하게 하는 핵심 공간.

---

## 공간 목표

- 사용자가 원하는 키워드를 직접 선택/조합
- 슬롯머신 형태의 재미있는 조합 경험
- AI 기반 아이디어 생성 및 비교
- Lite와 Pro의 핵심 가치를 체감하는 공간

---

## 핵심 기능

### 키워드 커스텀 선택 (Lite/Pro)

- Lite는 비AI 5개 카테고리에서 원하는 키워드를 직접 선택
- Pro는 AI를 포함한 전체 6개 카테고리에서 직접 선택
- 티어에 맞는 범위 안에서 자유 조합

### 슬롯머신 조합

- 게임적 UI로 키워드 조합의 재미 강화
- 랜덤 + 부분 고정 조합 지원

### AI 아이디어 생성

- 조합 기반 아이디어 draft 10개 생성
- 결과 비교 및 선택

### 커스텀 키워드 관리 (Lite/Pro)

- Lite는 비AI 5개 카테고리 기준으로 직접 키워드 추가/관리
- Pro는 AI 카테고리까지 포함해 전체 6개 카테고리 관리
- 자신만의 관심 영역 반영

### 프로젝트 개요 생성

- 선택한 아이디어 1개에 대해 상세 개요서 생성
- 문제 정의, 타겟, 핵심 기능, BM 구조 등 포함

### 감정 (Appraisal)

- 생성된 개요서의 시장성/실현성 기본 평가
- 정밀 감정은 Phase 2에서 확장

---

## 실험실 파이프라인

```
빠른 이해 → 입체적 판단 → 방향 선택 → MVP 설계 → 단계별 확장
```

### 1단계. 프로젝트 개요

**역할:** "이 원석이 어떤 프로젝트가 될 수 있는지 빠르게 이해하는 단계"

- **밀도:** 낮음~중간. 각 섹션 3~5줄. 1~2분 안에 전체 그림 파악.
- **느낌:** "기획 초안" — 발표 자료는 아니지만 "이거 괜찮은데?"를 느끼게.

**구성:**
- 문제 정의
- 타겟 유저
- 핵심 기능
- 차별점
- BM 초안
- MVP 범위

여기서 너무 깊게 들어가면 감정 단계와 MVP 청사진의 의미가 약해짐.

### 2단계. 감정

**역할:** "이 프로젝트를 여러 관점으로 읽어주는 단계"

- **밀도:** 낮음. 관점당 2~4문장, 축 4~6개. 1분 내외.
- **느낌:** 점수표가 아니라 **짧고 날카로운 코멘트 묶음.**

**추천 축:**
- 시장성
- 문제 적합성
- 실행 가능성
- 차별화 가능성
- 확장성
- 리스크

**예시:**
```
시장성: 수요 공감은 쉽지만 지불 의사로 연결되려면 기능 선명도가 더 필요함
실행 가능성: 현재 기술로 MVP 구현은 가능하지만 차별화 포인트가 약해질 수 있음
```

감정은 길게 읽는 단계가 아니라 "강점/약점을 빠르게 입체적으로 보는 단계".

### 3단계. 방향 정리

**역할:** "그래서 이 프로젝트를 어느 방향으로 밀어야 하는지 정리하는 단계"

- **밀도:** 중간. 항목당 2~3줄, 총 4~6개 제안. 1~2분.
- **느낌:** "코치의 조언" — 비판보다 판단을 도와주는 정리.

**추천 구조:**
- 더 좁혀야 할 타겟
- 더 강조해야 할 가치
- 버려도 되는 요소
- 먼저 검증해야 할 가설
- BM 조정 제안
- MVP 축소 제안

**예시:**
```
타겟은 '1인 가구 전체'보다 '야간 고립감을 자주 느끼는 20~30대 직장인'으로
좁히는 편이 MVP 설계에 유리함

감성 케어 전반보다 '밤 시간 체크인 루틴' 하나에 집중하면
초기 차별화가 선명해짐
```

### 4단계. MVP 청사진

**역할:** "실제로 첫 버전을 만든다면 어디까지를 MVP로 볼지 설계하는 단계"

- **밀도:** 높음. 각 섹션 4~8줄, 기능은 리스트형. 3~5분.
- **느낌:** "아이디어 설명"이 아니라 **제품 초안 설계 문서.**

**추천 구조:**
- MVP 목표
- 핵심 타겟
- 반드시 필요한 기능 (Must / Should / Later 우선순위)
- 제외할 기능
- 첫 검증 포인트
- 추천 사용자 플로우
- 성공 기준 초안

**예시:**
```
Must: 밤 시간 체크인, 대화 기록, 간단한 감정 상태 저장
Should: 주간 요약, 루틴 추천
Later: 전문가 연결, 멀티모달 입력, 커뮤니티 기능
```

MVP 청사진은 Pro 유료 가치의 핵심. "이건 실제로 일에 쓸 수 있다"고 느껴야 함.

### 5단계. Phase 로드맵

**역할:** "이 프로젝트를 단계적으로 어떻게 키울지 보여주는 단계"

- **밀도:** 중간~높음. 각 Phase 3~6줄, 총 3개 Phase. 2~4분.
- **느낌:** 일정표가 아니라 **의사결정의 흐름이 보이는 전략 문서.**

**추천 구조:**
- Phase 1: 가장 작은 검증 버전
- Phase 2: 리텐션/유료화 강화
- Phase 3: 확장/차별화/스케일

**예시:**
```
Phase 1: 밤 시간 체크인 중심의 가장 작은 감정 루틴 도구 출시
Phase 2: 개인화 추천과 유료 기능 추가
Phase 3: 주거 서비스/복지/B2B 확장 검토
```

## 단계별 정보 밀도 요약

| 단계 | 밀도 | 목적 | 읽는 시간 |
|------|------|------|----------|
| 프로젝트 개요 | 낮음~중간 | 빠른 이해 | 1~2분 |
| 감정 | 낮음 | 입체적 판단 | 1분 내외 |
| 방향 정리 | 중간 | 다음 선택 결정 | 1~2분 |
| MVP 청사진 | 높음 | 실제 MVP 설계 | 3~5분 |
| Phase 로드맵 | 중간~높음 | 단계별 실행 그림 | 2~4분 |

## 티어별 파이프라인 깊이

### Free

| 단계 | 접근 | 깊이 |
|------|------|------|
| 프로젝트 개요 | 1회/일 | 충분히 좋게 — 제품 실력 증명 |
| 기본 감정 | 1회 | 축소판 — 주요 축 점수만 |
| 방향 정리 | — | 짧게 한두 줄 수준 |
| MVP 청사진 | — | 잠금 |
| Phase 로드맵 | — | 잠금 |

### Lite

| 단계 | 접근 | 깊이 |
|------|------|------|
| 프로젝트 개요 | 여러 번 | 풀 |
| 기본 감정 | 넉넉 | 풀 |
| 정밀 감정 | 일부 | 얕은 — 점수 + 한 줄 코멘트 |
| 방향 정리 | 가능 | 풀 |
| 실행 설계 | 맛보기 | 요약 3줄 — "더 깊게 보고 싶다"를 유발 |
| MVP 청사진 | — | 미리보기 수준 |
| Phase 로드맵 | — | 잠금 |

### Pro

| 단계 | 접근 | 깊이 |
|------|------|------|
| 프로젝트 개요 | 거의 무제한 | 풀 |
| 기본 감정 | 충분 | 풀 |
| 정밀 감정 | 충분 | 풀 — 근거 + 유사 사례 + 리스크 |
| 심층 감정 리포트 | 가능 | 풀 |
| 방향 정리 | 가능 | 풀 + 상세 |
| 실행 설계 | 가능 | 풀 — 기능 목록 + 기술 스택 제안 |
| MVP 청사진 | 가능 | 풀 — Must/Should/Later + 검증 포인트 + 성공 기준 |
| Phase 로드맵 | 가능 | 풀 — 3단계 전략 + 각 Phase 목표/핵심 기능 |
| 비교/분기/재정제 | 가능 | 풀 |

### 깊이 레이어 핵심 원칙

Lite에서 Pro 기능을 "맛보기"로 보여줄 때, "부족함"이 아니라 **"더 깊게 보고 싶음"**을 만들어야 함.

**정밀 감정 차이 예시:**
```
Lite: "시장성: 7.2/10 — 헬스케어 AI 시장이 성장 중입니다"
Pro:  "시장성: 7.2/10 — 헬스케어 AI 시장이 연 23% 성장 중.
       유사 서비스 3개 분석: A사($2M 펀딩), B사(MAU 5만).
       차별점: 음성 AI 특화. 리스크: 규제 환경 변화"
```

**실행 설계 차이 예시:**
```
Lite: "이 아이디어는 모바일 앱 형태가 적합하며, MVP는 핵심 기능 3개로
       시작하는 것을 추천합니다"
Pro:  "MVP 기능 목록:
       (1) 음성 감정 입력 (2) AI 분석 리포트 (3) 주간 트래킹.
       기술 스택 제안: React Native + Whisper API + Supabase.
       Phase 1 범위: 4주, Phase 2: 감정 트렌드 대시보드"
```

## 풀 개요서 (Pro 전용 — AI 코딩용 내보내기)

파이프라인 전체(개요→감정→방향→청사진→로드맵)를 완료한 후, **"풀 개요서 내보내기"**로 하나의 구현 가능한 문서로 합쳐서 export. `/docs`에 넣으면 AI 코딩 도구(Claude Code, Cursor 등)가 바로 구현 시작할 수 있는 수준.

### 2종류 개요서

| | 라이트 개요서 | 풀 개요서 |
|--|------------|---------|
| 티어 | Free/Lite/Pro | **Pro 전용** |
| 목적 | 빠른 이해 (1~2분) | AI 코딩 도구가 바로 구현 시작 |
| 밀도 | 기획 초안 | 실행 가능한 기술 문서 |
| 생성 시점 | 원석 선택 직후 | 파이프라인 전체 완료 후 내보내기 |

### 풀 개요서 구성 (4블록 13섹션)

**── 비전 (왜 만드는가) ──**

| # | 섹션 | 신뢰도 |
|---|------|--------|
| 1 | 한 줄 컨셉 | 그대로 사용 가능 |
| 2 | 문제 정의 | 그대로 사용 가능 |
| 3 | 타겟 유저 | 그대로 사용 가능 |

**── 제품 (뭘 만드는가) ──**

| # | 섹션 | 신뢰도 |
|---|------|--------|
| 4 | 핵심 기능 (Must / Should / Later) | 시작점으로 사용, 검토 필요 |
| 5 | 유저 플로우 (step-by-step) | 시작점으로 사용, 검토 필요 (Happy path만, 엣지 케이스 빠짐) |
| 6 | 화면 목록 (이름 + 1줄 설명) | 시작점으로 사용, 검토 필요 |

**── 기술 (어떻게 만드는가) ──**

| # | 섹션 | 신뢰도 |
|---|------|--------|
| 7 | 기술 스택 | 시작점으로 사용, 검토 필요 (버전 호환성 검증 필요) |
| 8 | 데이터 모델 (테이블/스키마) | 반드시 검토/수정 후 사용 (관계, 인덱싱 확인 필요) |
| 9 | API 엔드포인트 | 반드시 검토/수정 후 사용 (스키마가 데이터 모델과 맞는지) |
| 10 | 파일 구조 | 반드시 검토/수정 후 사용 (프레임워크 convention 확인) |
| 11 | 필요한 외부 서비스 / API 키 | 그대로 사용 가능 |
| 12 | 인증 플로우 | 시작점으로 사용, 검토 필요 |

**── 비즈니스 (어떻게 굴리는가) ──**

| # | 섹션 | 신뢰도 |
|---|------|--------|
| 13 | BM/가격 + 핵심 비즈니스 규칙 + MVP 범위(IN/OUT) + 검증 질문 | 그대로 사용 가능 (가격은 가설) |

### 신뢰도 라벨 시스템

풀 개요서의 각 섹션에 신뢰도 라벨을 붙여서, AI 코딩 도구나 사용자가 맹신하지 않도록 안내:

```
[✅ 그대로 사용 가능]      — 비전/비즈니스 블록. 방향성 정보라 정확도 높음.
[🔍 시작점, 검토 필요]     — 제품/기술 블록 일부. 70~80% 정확. 엣지 케이스 보완 필요.
[⚠️ 반드시 검토 후 사용]   — 데이터 모델, API, 파일 구조. 60~65% 정확. 초안으로만.
```

### 불포함 항목 (의도적으로 뺌)

| 항목 | 불포함 이유 |
|------|-----------|
| 성공 기준 (숫자) | LLM이 근거 없이 찍음. "검증 질문"으로 대체 (13번에 포함) |
| 환경 설정 가이드 | 기술 스택(7번)이 있으면 AI 코딩 도구가 알아서 셋업 |
| AI 프롬프트 초안 | 정확도 50%. 잘못된 프롬프트가 더 해로움. 개발하면서 만드는 게 나음 |
| 차별점 | 라이트 개요서에만 포함. 구현에 직접 필요하지 않음 |
