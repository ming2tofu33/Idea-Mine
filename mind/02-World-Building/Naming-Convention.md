# Naming Convention

> 제품 내 모든 용어의 네이밍 원칙과 확정 용어 사전. TERMS.md의 원본.

---

## 네이밍 3원칙

| 원칙 | 적용 대상 | 예시 |
|------|----------|------|
| **세계관형** | 공간 이름 | 광산, 금고, 실험실 |
| **실무형** | 문서/결과물 이름 | 프로젝트 개요, MVP 청사진, 실행 설계 |
| **게임형** | 과금/확장 이름 | 야간 작업권, 정밀 감정권, 개요 확장권 |

---

## 1. 공간 용어

> 광산 / 금고 / 실험실은 반드시 메인 용어로 고정.

| 메인 용어 | 서브표기 | 역할 | 비고 |
|----------|---------|------|------|
| 광산 | The Mine | 홈, 키워드 발견/채굴 | 메인 탭 이름 |
| 금고 | The Vault | 저장 공간 | 메인 탭 이름 |
| 실험실 | The Lab | 문서화/정제 공간 | 메인 탭 이름 |
| 쇼케이스 | (영어 그대로) | 공개 전시 공간 | Phase 3+ |
| 거래소 | - | 투자자 연결/매칭 공간 | Phase 3+, 서브표기 불필요 |
| 전망대 | - | 분석/관측 공간 | Phase 3+, Observatory 굳이 안 붙임 |

## 2. 핵심 오브젝트 용어

### 광맥 계열

| 용어 | 의미 | 사용 맥락 |
|------|------|----------|
| 광맥 | 키워드 조합 단위 | 코어 도메인 모델 |
| 오늘의 광맥 | 홈 추천 조합 | 홈 화면 메인 |
| 특별 광맥 | 이벤트형 조합 | 주말/시즌 이벤트 |
| 취향 광맥 | 개인화된 추천 조합 | 유료 해금 |
| 희귀 광맥 | rarity 높은 광맥 | 희소성 연출 |
| 반짝 광맥 | 희귀도 중간 | 감성 표현 |

### 원석 계열

| 용어 | 의미 | 사용 맥락 |
|------|------|----------|
| 아이디어 원석 | 1차 발산 결과물의 공식명 | 정식 명칭 |
| 원석 | 줄임말 | UI 안에서 |

### 저장 관련

| 용어 | 의미 | 사용 맥락 |
|------|------|----------|
| 금고 반입 | 원석을 저장하는 행위 | 행동 버튼 |
| 반입 슬롯 | 한 번의 채굴에서 금고로 가져갈 수 있는 수 | 시스템 |

## 3. 문서/정리 단계 용어

> 유저가 바로 이해해야 하므로 실무형 원칙 적용.

| 용어 | 설명 | 접근 |
|------|------|------|
| 프로젝트 개요 | 무료에서 만드는 기본 문서 | 무료 |
| 감정 | 기본 평가/해석 | 무료 |
| 정밀 감정 | 유료 심화 평가 | 유료 |
| 실행 설계 | 프로젝트 실행 방향 정리 | 유료 |
| MVP 청사진 | MVP 단위까지 구체화된 설계 | 유료 |
| Phase 로드맵 | 단계별 추진 계획 | 유료 |

### "정제"의 위치

- **내부 개념:** 정제 (운영 문서, 기획 문서에서 사용)
- **유저 노출:** 실행 설계 / MVP 청사진 / 방향 조정 (UI에서는 구체 기능명)
- 정제는 유저에게 메인 기능명으로 내세우기엔 추상적 → 내부 용어로만

## 4. 행동 용어

> 버튼/CTA에 직접 쓰이는 단어.

### 기본 행동

| 용어 | 의미 |
|------|------|
| 채굴하기 | 가장 기본 행동 |
| 다시 파기 | 리롤 |
| 금고에 반입 | 저장 |
| 실험실로 보내기 | 다음 단계 이동 |
| 개요 만들기 | 프로젝트 개요 생성 |
| 감정하기 | 기본 평가 |
| 더 깊게 보기 | 유료 심화 진입용 보조 표현 |

### 버튼 텍스트 추천

| 버튼 | 맥락 |
|------|------|
| 이 광맥 파기 | 광맥 선택 |
| 원석 보기 | 결과 확인 |
| 금고에 반입 | 저장 |
| 실험실로 보내기 | 정제 시작 |
| 개요 만들기 | 문서 생성 |
| 정밀 감정 열기 | 유료 심화 |
| MVP 청사진 보기 | 유료 문서 |

> 게임형 원칙 적용. 보상/해금 느낌.

| 용어 | 설명 | 우선순위 |
|------|------|---------|
| 야간 작업권 | 추가 채굴/추가 사용 | 핵심 |
| 개요 확장권 | 개요 문서 추가 생성 | 핵심 |
| 정밀 감정권 | 심화 감정 추가 | 핵심 |
| 실험실 패스 | 실험실 유료 확장 접근 | 구독 혜택/번들용 |
| 청사진 확장권 | 실행 설계/MVP 청사진 확장 | 보조 |

## 8. 키워드 카테고리 노출명

> 내부 스키마명은 유지하고, 사용자에게는 더 자연스러운 라벨로 노출한다.

### 원칙

- 내부 스키마: `AI / Who / Domain / Tech / Value / Money`
- 사용자 노출(한국어): `AI / 대상 / 산업 / 형태 / 가치 / 수익모델`
- 사용자 노출(영어): `AI / Target / Industry / Platform / Value / Business Model`

### 카테고리 매핑

| 내부 스키마명 | 사용자 노출명 (KO) | 사용자 노출명 (EN) | 비고 |
|--------------|--------------------|--------------------|------|
| AI | AI | AI | AI 기술 자체를 뜻하므로 그대로 유지 |
| Who | 대상 | Target | "누구를 위한가"를 더 직관적으로 표현 |
| Domain | 산업 | Industry | 일반 사용자에게 Domain보다 명확 |
| Tech | 형태 | Platform | 구현 채널/제품 형태를 함께 포괄 |
| Value | 가치 | Value | 의미가 비교적 명확하므로 유지 |
| Money | 수익모델 | Business Model | Money보다 의도가 분명함 |

### 사용 원칙

- 기획/DB/프롬프트/운영 문서에서는 내부 스키마명을 사용해도 된다
- 앱 UI, 온보딩, 마케팅, 사용자 설명 문구에서는 사용자 노출명을 우선 사용한다
- `Who / Domain / Tech / Value / Money`를 그대로 노출하는 것은 지양한다
- 처음 3회 사용까지는 필요하면 사용자 노출명 뒤에 보조 설명을 붙일 수 있다
  - 예: `형태 (앱, 웹, 챗봇)` / `수익모델 (구독, 광고, B2B)`
