# Abuse Prevention & Rate Limiting Design

> 레이어드 방어 시스템. 봇 악용으로부터 API 비용, 콘텐츠, 서비스 품질을 보호한다.

---

## 배경

- 유료 유저(특히 Pro "사실상 무제한")가 봇으로 대량 호출하는 시나리오 방어
- 방어 대상: API 비용 폭탄 + 콘텐츠 스크래핑 + 서비스 품질 저하
- 미래에 MCP(Model Context Protocol) 프로그래밍 접근 가능성 고려
- `Clean-Mine-Protocol`은 콘텐츠 오염 방지 — 이 문서는 **사용량/비용 방어**에 집중

---

## 아키텍처: 4-Layer Defense

```
요청 → [L1 속도 제한] → [L2 일일 상한] → [L3 행동 패턴] → [L4 비용 차단기] → AI 호출
                                                                        ↓
                                                              [AI 비용 로깅]
```

각 레이어가 다른 종류의 악용을 잡는다. 정상 유저는 L1~L2에 절대 걸리지 않는다.

---

## L1. 속도 제한 (Rate Limiting)

봇의 첫 번째 특징: 사람보다 빠르다.

### 제한 기준

| 윈도우 | 제한 | 근거 |
|--------|------|------|
| 분당 | 3회 | 키워드 선택 → 결과 읽기 → 다음 채굴까지 최소 20초 |
| 시간당 | 20회 | 집중 파워유저도 이 안에 들어옴 |

### 초과 시 대응

- 즉시 차단이 아니라 **지연 응답** (429 + Retry-After)
- 세계관 메시지: "광맥이 불안정합니다. 잠시 후 다시 시도해주세요"
- 초과 누적 시 L3로 에스컬레이션

### 구현

- 유저 ID 기준 (IP 아님 — 로그인 필수 서비스)
- Supabase Edge Function 또는 Python 백엔드 미들웨어

---

## L2. 일일 상한 (Daily Cap)

"사실상 무제한" 셀링포인트를 유지하면서 비용 방어선 확보.

### 티어별 채굴 상한

| 티어 | 일일 상한 | 근거 |
|------|----------|------|
| Free | 1회 | 체험용 (기존 기획 유지) |
| Lite | 5회 | 실속형 (기존 기획 유지) |
| Pro | **50회** | 정상 파워유저(10~15회)의 3배+ 여유 |

- Pro 50회 = 하루 500개 아이디어 생성 가능 — 정상 사용에서 절대 도달 안 함
- 봇이 50회를 소진해도 L1과 결합하면 최소 2.5시간 소요

### 표시 방식

- 남은 횟수를 적극 노출하지 않음 — 정상 유저는 상한 존재를 모름
- 상한 도달 시에만: "오늘의 채굴 에너지를 모두 사용했습니다. 내일 광맥이 새로 열립니다"

---

## L3. 행동 패턴 분석 (Behavioral Detection)

L1~L2를 통과하는 정교한 봇을 잡는 레이어.

### 탐지 시그널

| 시그널 | 사람 | 봇 |
|--------|------|-----|
| 요청 간격 | 불규칙 (20초~5분) | 일정 (정확히 21초마다 등) |
| 결과 소비 시간 | 아이디어 읽고 저장/스킵 (30초+) | 생성 즉시 다음 요청 (0~2초) |
| 금고 반입률 | 10개 중 1~3개 저장 | 0개 저장 또는 전부 저장 |
| 세션 패턴 | 짧은 집중 후 이탈 | 시간 내내 일정한 호출 |
| 키워드 선택 | 고민하며 변경 | 랜덤 또는 고정 반복 |

### 의심 점수 (Suspicion Score)

- 각 시그널에 가중치 부여 → 내부 점수 누적
- 유저에게 점수는 비공개

### 의심 시 대응 (단계적)

| 점수 구간 | 대응 |
|----------|------|
| 낮음 | 로그만 기록 |
| 중간 | 응답 지연 (2~5초 추가) — 봇에겐 효율 저하, 사람은 못 느낌 |
| 높음 | "광산 안전 점검 중입니다" + 간단한 인터랙션 요구 |
| 매우 높음 | 세션 일시 중단 + 관리자 알림 |

### 구현 시점

- **Phase 1:** 시그널 로깅만 (데이터 수집)
- **Phase 1.5~2:** 실제 자동 대응 활성화 (수집된 데이터 기반 임계값 설정)

---

## L4. 비용 차단기 (Cost Circuit Breaker)

최후의 보험. L1~L3을 모두 뚫는 시나리오에서도 비용을 예측 가능한 범위로 제한.

### AI 호출 비용 맵

| 기능 | 비용 무게 |
|------|----------|
| 채굴 (아이디어 10개) | 중 |
| 프로젝트 개요 생성 | 대 |
| 기본 감정 | 중 |
| 정밀 감정 (Lite/Pro) | 대 |
| 심층 감정 리포트 (Pro) | 최대 |
| 실행 설계 (Pro) | 대 |
| MVP 청사진 (Pro) | 최대 |
| AI 키워드 보정 | 소 |

### 두 가지 차단기

| 차단기 | 기준 | 작동 |
|--------|------|------|
| 유저 단위 | 유저 1명의 일일 총 AI 토큰 비용 > 상한 | 해당 유저만 차단 |
| 시스템 전체 | 전체 일일 비용 > 예산 상한 | 신규 생성 일시 중단 + 관리자 긴급 알림 |

- 채굴 횟수가 아니라 **모든 AI 호출의 토큰 비용 합산** 기준
- 시스템 차단: 일일 예산 80% → 경고, 100% → 신규 생성 중단 (기존 데이터 열람은 가능)

### 유저 경험

- 유저 단위: "오늘의 광산 자원이 소진되었습니다"
- 시스템 단위: "광산 점검 중입니다. 잠시 후 다시 열립니다"

## AI 비용 로깅 시스템

방어 판단 근거 + 비즈니스 의사결정 데이터.

### 기록 필드 (13개)

| 필드 | 설명 | 용도 |
|------|------|------|
| user_id | 누가 | 유저별 비용 분석 |
| tier | 어떤 티어 | 티어별 수익성 분석 |
| session_id | 세션 식별 | 채굴→개요→감정 흐름 추적, 이탈 분석 |
| feature_type | 채굴/개요/감정/설계 등 | 기능별 비용 비교 |
| model | gpt-4o / gpt-4o-mini 등 | 모델별 비용 최적화 |
| prompt_version | 프롬프트 버전 (v1, v2...) | 프롬프트 튜닝 A/B 비교 |
| input_tokens | 입력 토큰 | 프롬프트 최적화 |
| output_tokens | 출력 토큰 | 출력 길이 조절 |
| total_cost_usd | 환산 비용 | 핵심 지표 |
| response_time_ms | 응답 시간 | 성능 모니터링 |
| status | success / error / filtered | 에러율 + Clean Mine 차단 비용 파악 |
| language | 한/영 | 언어별 비용 차이 |
| created_at | 타임스탬프 | 시간대별 분석 |

### 비즈니스 분석 활용

| 질문 | 데이터 |
|------|--------|
| Pro 유저 1명당 월 AI 비용? | user_id + tier + total_cost_usd |
| 어떤 기능이 가장 비싼가? | feature_type별 합산 |
| Pro $12.99가 수익이 나는가? | 티어별 ARPU vs AI 비용 |
| 프롬프트 개선 후 비용 줄었나? | prompt_version + 시점별 토큰 추이 |
| gpt-4o-mini로 전환해도 되나? | model별 비용/품질 비교 |
| 한국어가 영어보다 비싼가? | language별 토큰 비교 |
| 유저가 어디서 이탈하는가? | session_id로 흐름 추적 |

### 구현

- Supabase `ai_usage_logs` 테이블
- 모든 OpenAI 호출 후 동기적 기록 (비동기면 유실 위험)
- Phase 1부터 바로 적재

---

## 구현 타임라인

| Phase | 구현 범위 |
|-------|----------|
| Phase 1 (Sprint 2) | L1 속도 제한 + L2 일일 상한 + AI 비용 로깅 + L4 비용 차단기 |
| Phase 1 (Sprint 2) | L3 시그널 로깅 시작 (대응 없이 데이터 수집만) |
| Phase 1.5~2 | L3 자동 대응 활성화 (수집 데이터 기반 임계값) |
| MCP 오픈 시 | API Key 기반 별도 rate limit + 토큰 버킷 방식 검토 |

---

## `Clean-Mine-Protocol`과의 관계

| 시스템 | 역할 |
|--------|------|
| Clean Mine Protocol | **콘텐츠** 오염 방지 (음란/혐오/스팸 필터링) |
| Abuse Prevention (이 문서) | **사용량/비용** 악용 방지 (봇/대량호출 방어) |

두 시스템은 독립적으로 작동하되, 둘 다 위반하는 케이스(봇이 부적절 키워드를 반복 입력)는 양쪽에서 동시에 잡힘.
